## 1. 구현 요약

Phase15의 `FastAPI + React/Vite` MVP 구현을 진행했다.

이번 구현은 기존 파서 코어를 직접 수정하지 않고, 기존 CLI를 로컬 API에서 subprocess로 감싸는 방식으로 구성했다.

구현 범위:

- FastAPI API 어댑터
- in-memory Job Manager
- 업로드 파일 저장
- CLI subprocess 실행 명령 생성
- 작업 상태/취소/로그 tail 조회
- artifact 목록/다운로드
- `OUTPUT_QA_REPORT.md`와 `RUN_MANIFEST.json` 기반 QA 요약
- React/Vite 업무형 UI
- Google Drive npm 설치 우회 스크립트
- API focused tests

## 3. API 구현 내용

구현 endpoint:

```text
GET  /api/config
POST /api/jobs
GET  /api/jobs/{job_id}
POST /api/jobs/{job_id}/cancel
GET  /api/jobs/{job_id}/artifacts
GET  /api/jobs/{job_id}/artifacts/{artifact_id}
GET  /api/jobs/{job_id}/qa
```

작업 폴더:

```text
output/ui_runs/<job_id>/
  input/
  result/
  logs/
```

상태값:

```text
queued
running
succeeded
failed
canceled
```

보안 처리:

- 업로드 확장자 `.pdf` 제한
- 업로드 파일명 sanitize
- artifact id는 relative path를 서버가 인코딩해서 생성
- 다운로드는 result directory 내부만 허용
- `.env`, `.py`, `.db`, `.log` 다운로드 차단

## 5. 구현 중 발견한 문제와 반영 사항

### 5.1 Google Drive npm 설치 문제

문제:

```text
npm warn tar TAR_ENTRY_ERROR UNKNOWN: unknown error, write
npm error EBADF: bad file descriptor, write
```

원인 판단:

- Google Drive 동기화 경로 안에 `frontend/node_modules`를 직접 설치하면서 npm tar 추출과 Drive 파일 동기화/잠금이 충돌했다.
- `lucide-react` 제거 후에도 Vite/Babel 계열 패키지 설치에서 같은 문제가 반복되어, 특정 패키지 문제가 아니라 Drive 경로의 대량 파일 쓰기 문제로 판단했다.

반영:

- `scripts/frontend_local.ps1` 추가
- 프론트 소스를 `%LOCALAPPDATA%\Codex\ps-docparser-frontend-work`로 복사
- npm install/build/dev를 Drive 밖에서 실행
- `.gitignore`에 `frontend/node_modules/`, `frontend/node_modules.partial-delete*/`, `frontend/dist/` 추가

### 5.2 pytest api marker 스킵 문제

문제:

- `tests/unit/api` 폴더명이 기존 `conftest.py`의 `"api" in item.keywords` 조건에 걸려 API unit test가 모두 skip되었다.

반영:

- 실제 `@pytest.mark.api`가 붙은 테스트만 `--run-api` 없을 때 skip되도록 `item.get_closest_marker("api")` 기준으로 수정했다.
- `slow` marker도 같은 방식으로 좁혔다.

### 5.3 icon dependency 조정

문제:

- Google Drive npm 설치 안정성 확보가 우선이었다.

반영:

- v1에서는 `lucide-react` 의존성을 제외했다.
- UI는 CSS 기반 `icon-mark` 보조 표식을 사용한다.
- 비동기 설치 환경이 안정된 로컬/CI에서는 추후 lucide icon으로 교체 가능하다.

## 7. 남은 위험

- 실제 PDF 업로드 E2E는 API/UI 구조와 build 검증까지 완료했지만, 장시간 OCR 엔진 실행은 이번 구현 검증에서 제외했다.
- `output/ui_runs/`는 Git 추적 대상이 아니며, 실제 실행 후 산출물 보존/삭제 정책을 커밋 전 다시 확인해야 한다.
- 서버 재시작 시 in-memory job registry는 사라진다. v1 MVP에서는 허용하지만, 이후 운영형으로 가려면 job metadata 파일 또는 DB가 필요하다.
- Google Drive 경로에서는 앞으로도 `node_modules`를 직접 만들지 않는 것이 안전하다.
