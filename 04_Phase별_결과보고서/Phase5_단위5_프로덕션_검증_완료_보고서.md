## 2. 단위 5 검증 중 발견된 치명적 결함(버그) 및 해결 내용
실제 운영 배치를 수행하는 과정에서 `ps-docparser` 내부 파이프라인의 내결함성(Fault-tolerance)을 해칠 수 있는 3가지 중대 문제를 발견하여 수정 완료했습니다.

### ① ZAI API 응답 JSON 구조 불안정 대처 (`AttributeError` 방어)
- **현상**: API 반환 배열 중 `layout_details`의 내부 아이템이 간헐적으로 `dict`가 아닌 `list` 형태로 주입되어, 파싱 과정 중 `item.get("label")` 시점에 파이프라인 전체가 크래쉬 됨.
- **조치**: `bom_extractor.py` 내부의 탐색 루프에 `if not isinstance(item, dict): continue` 방어 코드 삽입. (타입 세이프티 확보)

### ② JSON 인코딩 문제로 인한 파싱 크래쉬 대처
- **현상**: Python 엑셀 변환기가 부수적으로 내보낸 JSON 데이터가 BOM(Byte-Order-Mark)를 포함한 `utf-8-sig` 형식으로 저장됨. 그러나 집계 모듈(`bom_aggregator.py`)에서는 일반 `utf-8`로 읽으려 시도하다가 파싱 에러(Unexpected UTF-8 BOM) 발생.
- **조치**: 집계기의 파일 오픈 로직을 `with open(jpath, "r", encoding="utf-8-sig")`으로 변경하여 호환성 확보.

### ③ 유효하지 않은 메타데이터 행(Line List)이 집계에 오염되는 문제 해결
- **현상**: Line List 테이블의 일부 열 값(예: 450A)이 BOM의 SIZE 컬럼으로 잘못 취급되어, 수치와는 무관한 쓰레기 행이 결과 엑셀에 0건으로 포함되는 현상 발견.
- **조치**:
    1. 헤더명 매핑(Alias) 로직 검색 방향 역방향 호출 오동작 판별 및 O(1) 역방향 딕셔너리(`_ALIAS_TO_CANONICAL`) 생성 도입 (정교한 키 매핑 로직 구축).
    2. 데이터 유효성 검사 강화: SIZE/MATERIAL 컬럼 문자열이 존재하더라도, 추출된 최종 **Q'TY 와 WEIGHT 수치가 둘 다 `0`인 열은 BOM 데이터가 아니라고 판단하고 집계 합의 루프에서 즉시 제외(`continue`)**.
