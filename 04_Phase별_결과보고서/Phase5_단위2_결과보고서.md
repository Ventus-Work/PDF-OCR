# Phase 5: 단위 2 (배치 루프 및 캐시 주입 컨트롤러 리팩터링) 구현 결과 보고서

## 1. 개요
본 보고서는 `ps-docparser` 파이프라인의 핵심인 **Phase 5 매니저 모듈(Unit 2) 리팩터링 및 배치 통합 검증 결과**를 요약합니다.
이 과정을 통해 61개 이상의 대량 도면 PDF 폴더 전체를 한 번의 명령어 실행으로 순차 변환할 수 있도록 확장되었으며, 단위 1에서 구축한 SQLite 기반 **캐시 테이블 계층(TableCache)이 메인 엔진에 성공적으로 주입(Dependency Injection)** 되었습니다.

---

## 2. 주요 구현 내역 (코드베이스 수정)

| 생성/수정 파일 | 상세 내용 |
| --- | --- |
| **[NEW] `test_phase5_unit2.py`** | AST(추상 구문 트리) 파싱 기반의 메인 컨트롤러 독립 정적 검증 도구. API/외부 라이브러리 연결 없이도 아키텍처 규칙이 지켜졌는지 확인하는 테스트케이스(`TC-1` ~ `TC-10`) 작성. |
| **[MOD] `main.py`** | 575줄의 절차적 실행 구조를 전면 리팩터링. 비즈니스 로직에 관여하지 않는 컨트롤러의 역할을 충실히 따릅니다.<br>- **`ParserError` 파생 예외 적용**: 처리 중간에 발생하던 치명적 `sys.exit()` 강제 종료 지점 6곳을 걷어내고 사용자 정의 예외(`raise ParserError`)로 교체하여 개별 도면 오류가 전체 배치를 중단시키지 않게 설계했습니다.<br>- **`_process_single()` 파이프라인 분리**: 순수하게 단건 파일을 처리하던 로직 전체(프리셋, OCR 추출, JSON 내보내기, 엑셀 출력 등)를 독립 함수화하여 격리했습니다.<br>- **폴더 스캐닝/배치 루프 구현**: 파라미터가 디렉토리일 경우 해당 폴더 내의 `.pdf` 전체를 읽어들여 1번 객체부터 끝까지 순차 처리 및 성공/실패 목록을 요약 정리합니다.<br>- **캐시 주입**: 환경변수가 `CACHE_ENABLED=true` 인 경우에만 DB를 로딩, 캐시를 지참하여 AI 엔진(`bom_engine.cache = cache`)에 넘기는 안전한 초기화 흐름을 확보했습니다. |
