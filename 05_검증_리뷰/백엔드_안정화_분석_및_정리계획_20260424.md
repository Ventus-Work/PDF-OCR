# 백엔드 안정화 분석 및 정리 계획

작성일: 2026-04-24

대상:

- `ps-docparser`
- 실측 출력: `ps-docparser/output/실측테스트_20260424_151059`
- 관련 보고서: `ps-docparser/output/실측테스트_20260424_151059/OUTPUT_ANALYSIS_REPORT.md`

## 1. 현재 상태 요약

현재 백엔드는 “완전히 무너진 상태”라기보다, 기능이 빠르게 붙으면서 책임 경계가 흐려진 상태에 가깝다.

실측 테스트 기준으로 다음은 안정화되었다.

- 출력 폴더를 별도로 지정하면 결과가 분리된다.
- BOM primary와 fallback estimate가 서로 다른 파일로 저장된다.
- JSON `headers`와 row key 개수 불일치는 없다.
- 자기 반복 복합 헤더는 제거되었다.
- Excel은 JSON 구조를 그대로 따라간다.

그러나 다음 문제는 아직 남아 있다.

- BOM 첫 데이터 행 일부가 여전히 한 칸 밀릴 수 있다.
- BOM fallback estimate가 너무 쉽게 생성된다.
- 자동 라우팅, 명시 preset, fallback 정책이 서로 다른 기준으로 움직인다.
- JSON 스키마가 문서 유형별로 명확히 고정되어 있지 않다.
- Excel 시트 분류가 파서 결과를 보정하는 역할까지 일부 떠안고 있다.
- batch aggregation은 primary/fallback/line list 구분 정책이 아직 강하지 않다.
- 실측 테스트 결과를 자동으로 판정하는 검증 도구가 부족하다.

## 2. 불안정성이 보이는 실제 증거

### 2.1 260421 BOM 첫 행 열 밀림

실측 결과:

```text
DESCRIPTION = Scaffolding 설치
DWG NO. = KO-D-010-14-16N
수량 = ""
단위 = 1.0
자재중량 [Kg] | UNIT = 식
```

의도상 결과:

```text
DESCRIPTION = Scaffolding 설치
DWG NO. = KO-D-010-14-16N
수량 = 1.0
단위 = 식
자재중량 [Kg] | UNIT = ""
```

의미:

- 2행 헤더 제거와 복합 헤더 병합은 성공했다.
- 하지만 희소 데이터 행 정렬은 아직 완성되지 않았다.
- 특히 첫 행처럼 `SIZE`가 비고 `수량/단위`만 있는 행에서 밀림이 남는다.

### 2.2 PIPE 도면 fallback estimate 과다 생성

PIPE-FP-PS-5007~5013 primary BOM은 정상이다.

- `BOM_자재`: 1행
- `BOM_LINE_LIST`: 1행
- Excel: `BOM_자재표`, `BOM_자재표_2`

그러나 fallback estimate도 함께 생성된다.

fallback estimate에는 다음 노이즈가 들어간다.

- `TOTAL WEIGHT:44.41kg` 같은 합계 행이 모든 컬럼에 복제됨
- `LINE LIST`가 BOM row처럼 들어감
- 도면 revision block이 `D_기타` 표로 들어감

의미:

- fallback은 보조본으로는 유용할 수 있다.
- 하지만 정상 BOM 도면에서도 fallback이 자주 만들어지면 사용자가 대표 산출물을 헷갈릴 수 있다.

### 2.3 품셈 문서가 자동 실행에서는 범용 Table로 남음

`53-83 OKOK.pdf`는 문서 감지상 품셈 후보로 보이지만, 이번 실행은 범용 자동 파싱 결과로 저장되었다.

결과:

- `본문`
- `Table`
- `Table_2`
- ...
- `Table_11`

의미:

- 구조 추출 자체는 된다.
- 그러나 도메인별 최종 결과 품질은 preset을 명시했는지, 자동 라우팅이 실제로 전문 파이프라인을 탔는지에 따라 크게 달라진다.

### 2.4 거래명세서는 범용 테이블 수준

거래명세서는 표는 잡히지만 컬럼이 넓게 펼쳐진다.

예:

- `거래명세표(공급자보관용)_일련번호`
- `거래명세표(공급자보관용)_일련번호_2`
- ...

의미:

- 범용 추출로는 “읽을 수 있는 표”는 만들 수 있다.
- “거래명세서 업무 데이터”로 바로 쓰려면 별도 schema/preset이 필요하다.

## 3. 원인 분석

### 3.1 파서가 데이터 보정 책임을 너무 많이 가진다

현재 BOM HTML 파싱 흐름은 다음 일을 한 곳에서 처리한다.

- HTML table 추출
- BOM/LINE LIST 후보 판정
- title row 제거
- 2행 헤더 판정
- 복합 헤더 병합
- 희소 행 재정렬
- 노이즈 row 제거

이 구조는 빠르게 고치기에는 좋지만, 케이스가 늘어날수록 조건문이 서로 간섭한다.

대표적으로 `_realign_sparse_bom_row()`는 복합 헤더 구간 기준으로 row를 우측 정렬한다. 그런데 260421 첫 행처럼 실제로는 복합 헤더 구간이 아니라 일반 `수량/단위` 구간으로 값을 당겨야 하는 케이스가 남아 있다.

### 3.2 BOM 결과 schema가 아직 “계약”으로 고정되지 않았다

현재 BOM 변환기는 파서가 준 headers/rows를 받아서 dict로 만든다.

이때 다음은 보장되지만:

- header 중복 suffix
- row dict key 개수 정합성

다음은 아직 강하게 보장되지 않는다.

- `수량` 컬럼에는 숫자/수량이 들어간다.
- `단위` 컬럼에는 `식`, `EA`, `SET` 같은 단위가 들어간다.
- `자재중량 [Kg] | UNIT`에는 일반 단위가 들어가지 않는다.
- fallback estimate는 대표 BOM 집계 대상이 아니다.

즉 구조적 정합성은 잡혔지만 의미적 정합성은 아직 약하다.

### 3.3 자동 라우팅과 fallback 정책이 한 시스템처럼 보이지 않는다

현재 경로는 크게 세 가지다.

- 명시 preset: `--preset bom`, `--preset estimate`, `--preset pumsem`
- 자동 감지: preset 없이 실행 후 `detector.py`가 판단
- BOM 내부 fallback: `BomPipeline`이 estimate를 보조 실행

문제는 이 셋의 기준이 다르다는 점이다.

- 자동 감지는 문서 전체 키워드 중심이다.
- BOM fallback은 BOM 추출 결과의 table 수, 도면 메타데이터, raw text 길이 중심이다.
- Excel 분류는 table header와 일부 row 값 중심이다.

따라서 사용자는 “같은 PDF인데 실행 옵션에 따라 전혀 다른 결과”를 받는다.

### 3.4 Excel exporter가 downstream인데 일부 판정 책임을 가진다

Excel exporter는 원칙적으로 JSON을 보기 좋게 쓰는 downstream이어야 한다.

하지만 현재는 `_classify_table()`에서 다음 역할까지 한다.

- estimate/detail/condition/generic/BOM 분류
- BOM sheet 이름 결정의 출발점 제공
- generic table sheet 이름 결정

이 때문에 JSON 단계에서 문서 유형이 명확하지 않으면 Excel 단계가 추측으로 결과물을 만든다.

### 3.5 batch aggregation의 입력 선택 기준이 파일명 중심이다

batch BOM 집계는 성공한 PDF명 stem으로 `*_{stem}_bom.json`을 찾는다.

이 방식은 지금은 동작하지만 다음 위험이 있다.

- 같은 날짜에 같은 stem rerun이 여러 개 있으면 최신 후보를 선택한다.
- output 폴더에 유사 파일이 섞이면 대표본 선택이 흐려진다.
- fallback 파일은 `_bom_fallback_estimate.json`이라 일반적으로 제외되지만, 정책이 manifest로 고정된 것은 아니다.

## 4. 정리 방향

백엔드를 한 번에 갈아엎기보다, “계약을 세우고 그 계약 주변만 고정”하는 방식이 안전하다.

권장 방향은 다음 순서다.

1. BOM 의미 정합성 우선 고정
2. 실행 결과 manifest 도입
3. fallback 정책을 명시적 산출물 등급으로 분리
4. 도메인 preset별 출력 schema 정의
5. 실측 테스트 harness 자동화
6. exporter는 schema를 표현하는 역할로 축소
7. aggregation은 manifest와 schema 기준으로 입력 선택

## 5. Phase별 상세 정리 계획

### Phase A. BOM 첫 행 의미 정합성 패치

목표:

- 260421 첫 row의 `수량/단위` 밀림을 제거한다.
- 이미 정상인 2행 이후는 건드리지 않는다.

구현 방향:

- BOM row alignment helper를 복합 헤더 기준 정렬과 일반 수량/단위 보정으로 분리한다.
- 다음 조건을 만족할 때만 좁게 보정한다.

조건 예시:

```text
DESCRIPTION 있음
DWG NO. 있음
MAT'L 비어 있음
SIZE 비어 있음
수량 비어 있음
단위가 숫자형
다음 컬럼 값이 단위 문자열(EA, 식, SET 등)
```

보정:

```text
수량 <- 단위
단위 <- 다음 컬럼
다음 컬럼 <- ""
```

테스트:

- 260421 첫 row fixture
- 260421 2행 정상 row 무변경
- PIPE BOM row 무변경
- `headers == rows[0].keys()` 유지

완료 기준:

- 260421 첫 row가 `수량=1.0`, `단위=식`으로 나온다.
- 기존 복합 헤더 테스트가 모두 통과한다.

### Phase B. Run manifest 도입

목표:

- 어떤 파일이 primary인지, fallback인지, compare인지 명확히 기록한다.
- 사용자가 output 폴더에서 대표 산출물을 바로 알 수 있게 한다.

산출물:

```text
RUN_MANIFEST.json
RUN_SUMMARY.md
```

manifest 필드 예시:

```json
{
  "run_id": "실측테스트_20260424_151059",
  "created_at": "2026-04-24T15:10:59+09:00",
  "inputs": [
    {
      "source_pdf": "...",
      "preset": "bom",
      "engine": "zai",
      "primary": {
        "json": "..._bom.json",
        "xlsx": "..._bom.xlsx",
        "role": "representative"
      },
      "fallbacks": [
        {
          "json": "..._bom_fallback_estimate.json",
          "xlsx": "..._bom_fallback_estimate.xlsx",
          "role": "diagnostic"
        }
      ],
      "status": "success",
      "warnings": ["mixed_document"]
    }
  ]
}
```

완료 기준:

- 집계는 manifest의 `role=representative`만 사용한다.
- fallback은 diagnostic으로만 표시된다.
- report는 manifest를 읽어 자동 생성할 수 있다.

### Phase C. BOM fallback 정책 재정리

목표:

- fallback이 유용한 경우와 노이즈가 되는 경우를 분리한다.

현재 문제:

- 정상 PIPE 도면에서도 fallback이 만들어진다.
- fallback 결과가 primary보다 행 수가 많아 보여 사용자를 헷갈리게 한다.

권장 정책:

1. 기본값은 fallback 생성 유지
2. 단, primary BOM 품질이 충분하면 fallback은 `diagnostic`으로 명확히 표시
3. CLI에 선택지를 추가

CLI 옵션 제안:

```text
--bom-fallback auto      기본값, 의심 케이스만 생성
--bom-fallback always    항상 생성
--bom-fallback never     생성 안 함
```

품질 기준 예시:

```text
has_bom_rows >= 1
has_bom_core_headers >= 3
has_line_list_rows >= 0
has_drawing_meta or short_drawing_like_text
```

완료 기준:

- PIPE 도면은 primary가 정상일 때 fallback 과다 생성이 줄어든다.
- 혼합 견적/BOM 문서는 fallback이 계속 생성된다.
- README에 대표본/보조본 정책이 명확히 적힌다.

### Phase D. 도메인별 schema 계약 정의

목표:

- JSON이 downstream Excel/집계의 기준이 되도록 한다.
- “구조는 맞지만 의미가 틀린” 상태를 줄인다.

schema 파일 또는 dataclass 제안:

```text
schemas/
  bom_schema.py
  estimate_schema.py
  pumsem_schema.py
  trade_statement_schema.py
```

BOM schema 예시:

```text
BOM material table:
  required_any: DESCRIPTION or SIZE
  optional: DWG NO., MAT'L, QTY, UNIT, WEIGHT, REMARK
  invariant:
    QTY-like columns should not contain unit-only strings
    UNIT-like columns should not contain pure numeric quantity when QTY is empty
```

완료 기준:

- 파서 결과가 schema validator를 통과해야 대표본으로 표시된다.
- 실패 시 JSON은 저장하되 `quality.status=warning`으로 표시한다.

### Phase E. 실측 테스트 harness 자동화

목표:

- 지금 손으로 한 실측 분석을 반복 가능한 명령으로 만든다.

명령 제안:

```powershell
python tools/run_real_pdf_suite.py --case-set 20260424_bom_backend --engine zai
python tools/analyze_outputs.py ps-docparser/output/실측테스트_YYYYMMDD_HHMMSS
```

자동 검증 항목:

- JSON 개수
- Excel 개수
- header/key mismatch
- bad composite headers
- 첫 data row가 header가 아닌지
- 수량/단위 의미 정합성
- fallback과 primary 분리 여부
- 품셈 preset 실행 여부
- aggregation 입력 파일 목록

완료 기준:

- 실측 결과 폴더 하나를 넣으면 PASS/WARN/FAIL 보고서가 자동 생성된다.

### Phase F. Excel exporter 책임 축소

목표:

- Excel exporter는 “분류기”가 아니라 “렌더러”에 가깝게 만든다.

현재:

- Excel 단계에서 table을 보고 estimate/detail/condition/BOM을 추측한다.

권장:

- parser/converter가 table role을 확정한다.
- Excel exporter는 `table["role"]` 또는 `section["domain"]`을 우선 사용한다.
- 추측 분류는 마지막 fallback으로만 둔다.

필드 제안:

```json
{
  "domain": "bom",
  "role": "primary_material_table",
  "quality": {
    "status": "ok",
    "warnings": []
  }
}
```

완료 기준:

- BOM은 항상 `BOM_자재표` 계열로 간다.
- estimate는 항상 `견적서/내역서/조건` 계열로 간다.
- 분류 불가일 때만 `Table`로 간다.

### Phase G. aggregation 입력 정책 고정

목표:

- BOM batch 집계가 잘못된 파일을 먹지 않게 한다.

권장:

- `*_bom.json` 파일명 glob 대신 manifest를 읽는다.
- `role=representative`, `domain=bom`, `quality.status in ok/warning`만 집계한다.
- fallback estimate, compare baseline, diagnostic output은 제외한다.

완료 기준:

- PIPE 7개 집계에서 fallback row가 들어가지 않는다.
- LINE LIST는 별도 sheet로 보존된다.
- 집계 대상 파일 목록이 Excel 또는 summary에 기록된다.

## 6. 우선순위

### 즉시 해야 할 일

1. 260421 첫 row 수량/단위 보정
2. 실측 output analyzer를 테스트/도구로 고정
3. fallback 결과를 diagnostic으로 명확히 표시

### 다음 단계

1. `RUN_MANIFEST.json` 도입
2. aggregation 입력을 manifest 기준으로 변경
3. 품셈은 `--preset pumsem` 실측 세트 별도 구성

### 이후 단계

1. 도메인별 schema validator
2. Excel exporter 분류 책임 축소
3. 거래명세서 preset 여부 결정

## 7. 리스크 관리

### 피해야 할 접근

- BOM과 estimate 결과를 바로 merge하지 않는다.
- fallback estimate를 대표 산출물로 승격하지 않는다.
- Excel에서 보이는 모양만 맞추고 JSON 의미 오류를 방치하지 않는다.
- 모든 문서 유형을 하나의 generic parser heuristic으로 해결하려 하지 않는다.

### 안전한 접근

- primary/fallback/compare/diagnostic 역할을 먼저 분리한다.
- JSON 의미 정합성을 Excel보다 먼저 고친다.
- 실측 fixture를 대표 케이스로 고정한다.
- 라우팅 정책 변경은 작은 테스트 세트와 함께 진행한다.

## 8. 결론

현재 백엔드의 핵심 문제는 “기능 부족”보다 “결과물 역할과 schema 계약이 약한 것”이다.

가장 먼저 고쳐야 할 것은 260421 BOM 첫 행의 수량/단위 밀림이다. 그 다음에는 fallback과 primary를 시스템적으로 구분하는 manifest를 도입해야 한다. 이 두 가지가 잡히면 사용자가 결과 폴더를 열었을 때 무엇이 대표본이고 무엇이 보조본인지 명확해지고, batch aggregation도 안정적으로 정리할 수 있다.

## 9. 2026-04-24 구현 반영 현황

이번 실행에서는 위 계획 중 백엔드 안정화의 1차 기준선을 코드와 테스트에 반영했다.

### 9.1 Phase 1 - BOM 첫 행 의미 정합성

반영 내용:

- `parsers/bom_table_parser.py`에 `수량/단위` 한 칸 밀림 전용 보정 로직 추가
- 보정 조건을 좁게 제한
  - `수량` 계열 열은 비어 있음
  - `단위` 계열 열에 숫자형 값이 있음
  - 다음 UNIT 계열 열에 `식`, `EA`, `SET`, `LOT` 등 단위 값이 있음
- 조건이 맞을 때만 `수량 <- 단위`, `단위 <- 다음 UNIT`, `다음 UNIT <- ""`로 좌측 보정
- 정상 PIPE BOM row와 이미 정상인 row는 바꾸지 않는 회귀 테스트 추가

확인 기준:

- 260421 대표 fixture 첫 row가 `수량=1.0`, `단위=식`으로 고정됨
- `DESCRIPTION | DESCRIPTION`, `DWG NO. | DWG NO.` 자기 반복 헤더 재발 없음
- JSON header와 row key 순서 정합성 유지

### 9.2 Phase 2 - Quality validator와 role metadata

반영 내용:

- `validators/output_quality.py` 추가
- BOM table에 대해 아래 warning/status를 계산
  - `header_row_key_mismatch`
  - `self_repeating_composite_header`
  - `first_row_looks_like_header`
  - `qty_unit_shift_suspected`
  - `empty_tail_column`
- `extractors/bom_converter.py`가 BOM/LINE LIST table에 `domain`, `role`, `quality`를 부여

역할 기준:

- BOM 대표 자재표: `domain=bom`, `role=primary_material_table`
- LINE LIST: `domain=bom`, `role=line_list_table`
- validator는 데이터를 수정하지 않고 `quality.status`, `quality.warnings`만 기록

### 9.3 Phase 3 - fallback 정책과 manifest

반영 내용:

- CLI에 `--bom-fallback auto|always|never` 추가
- 기존 `--no-bom-fallback`은 호환용 alias로 유지
- 기본값은 `auto`
- `RUN_MANIFEST.json`, `RUN_SUMMARY.md` 생성 유틸리티 추가
- BOM primary 산출물은 `role=representative`, fallback estimate 산출물은 `role=diagnostic`으로 기록
- batch BOM aggregation이 manifest의 representative BOM JSON을 우선 사용하도록 변경

정책 기준:

- PIPE처럼 BOM/LINE LIST가 정상인 도면은 auto fallback을 줄임
- 혼합 견적/BOM처럼 fallback이 필요한 문서는 auto fallback 유지
- fallback estimate는 대표본으로 승격하지 않고 merge하지 않음

### 9.4 Phase 4 - Excel/집계/실측 검증 도구

반영 내용:

- Excel classifier가 `domain/role`을 우선 사용하도록 변경
- BOM material table은 `BOM_자재표`, LINE LIST는 `BOM_LINE_LIST` 계열 sheet로 렌더링
- legacy JSON은 기존 classifier fallback 유지
- `tools/analyze_outputs.py` 추가
  - output 폴더의 JSON/Excel/manifest 존재 여부 확인
  - header/key mismatch, bad composite header, quality warning 요약
  - `OUTPUT_QA_REPORT.md` 생성
- `tools/run_real_pdf_suite.py` 추가
  - 실측 PDF 세트를 timestamp output 폴더에 실행
  - 실행 후 analyzer를 호출해 QA report 생성
