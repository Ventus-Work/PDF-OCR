## 1. Executive Summary

P1~P4 패치는 **단위 테스트는 모두 통과**했고 P3·P4 같은 분류/경고 동작도 의도대로 작동했다.
그러나 **실제 PDF 1건(HD현대오일뱅크 10TON CRANE)을 돌려본 결과, P2(다단 헤더 병합)가 의도한 효과를 내지 못했다.**

핵심 결론은 4가지다.

1. P2는 단순 2행 헤더만 가정했지만, 실제 자재표는 **`rowspan=2` + `colspan=N`이 섞인 복합 구조**다.
2. expand_table 결과 1·2행에 동일 텍스트가 복제되는데, P2가 이를 그대로 ` | `로 이어붙여 **`DESCRIPTION | DESCRIPTION` 같은 의미 없는 중복 헤더**가 생성됐다.
3. 일부 데이터 행은 td 갯수가 헤더 컬럼 수보다 적어 **컬럼이 한 칸씩 밀려** 매핑됐다.
4. P4 혼합문서 경고는 정상 출력됐지만, **실제 내역서 표는 여전히 누락**되어 사용자는 결국 `--preset estimate`로 수동 재실행해야 했다.

한 줄 요약:

> **테스트는 통과했지만 테스트 입력이 실제 문서의 rowspan/colspan 혼합 패턴을 반영하지 않았다. 실측 케이스에서 P2는 무력했고, P4는 경고만 띄웠을 뿐 추출은 보강하지 못했다.**

## 4. P2 패치 결과 실측

### 4.1 JSON 출력 헤더 (`...10TON CRANE 설치_bom.json:13~30`)

```json
"headers": [
  "DESCRIPTION | DESCRIPTION",
  "DWG NO. | DWG NO.",
  "MAT'L | MAT'L",
  "SIZE | SIZE",
  "수량 | 수량",
  "단위 | 단위",
  "자재중량 [Kg] | UNIT",
  "자재중량 [Kg] | WEIGHT",
  "자재중량 [Kg] | LOSS",
  "자재중량 [Kg] | WEIGHT_2",
  "자재면적 [m2] | UNIT",
  "자재면적 [m2] | m2",
  "자재면적 [m2] | LOSS",
  "자재면적 [m2] | m2_2",
  "비고 | 비고",
  "비고"
]
```

### 4.2 무엇이 잘됐는가

- `자재중량 [Kg] | UNIT`, `자재중량 [Kg] | WEIGHT` 등 **진짜 2단 헤더는 의도대로 병합**됨
- P1 dedupe로 중복 `WEIGHT → WEIGHT_2`, `m2 → m2_2`로 분리
- 결과적으로 `UNIT/WEIGHT/LOSS/WEIGHT` 4개 보조열이 모두 살아남음

### 4.3 무엇이 잘못됐는가

- **`DESCRIPTION | DESCRIPTION`** 같은 rowspan=2 컬럼이 의미 없는 중복으로 표기됨 (6개 컬럼 해당)
- 마지막 컬럼에 `"비고 | 비고"`와 `"비고"` 두 개가 동시에 존재 → **헤더가 16개로 의도(15개)보다 1개 더 많음**
- 데이터 행 일부가 한 칸씩 밀려 들어감 (4.4 참조)

### 4.4 데이터 행 미스얼라인먼트

**원본 첫 데이터 행 HTML:**
```html
<tr>
  <td>Scaffolding 설치</td>
  <td>KO-D-010-14-16N</td>
  <td></td><td></td><td></td>
  <td>1.0</td><td>식</td>
</tr>
```

td 7개만 존재 (원본 테이블 의도는 15컬럼).

**JSON 출력 결과 (line 32~49):**
```json
{
  "DESCRIPTION | DESCRIPTION": "Scaffolding 설치",
  "DWG NO. | DWG NO.":         "KO-D-010-14-16N",
  "MAT'L | MAT'L":             "",
  "SIZE | SIZE":               "",
  "수량 | 수량":                 "",         ← 1.0 이어야 함
  "단위 | 단위":                 "1.0",      ← "식" 이어야 함
  "자재중량 [Kg] | UNIT":        "식",        ← 빈값 이어야 함
  ...
}
```

원본의 빈 td 3개(`<td></td><td></td><td></td>`)가 `MAT'L/SIZE/수량` 자리에 채워지고, 그 다음 `1.0`이 **단위** 컬럼으로, `식`이 **자재중량 UNIT** 컬럼으로 들어가 한 칸씩 밀렸다.

**비교: 두 번째 행 (`Travelling Rail 설치`)은 정상 매핑됨.**
```json
{
  "SIZE | SIZE":  "19,800M(6000=3,1800=1)x 2Set",
  "수량 | 수량":   "1.0",
  "단위 | 단위":   "식"
}
```
즉 td 갯수가 충분한 행은 정상이고, 부족한 행은 밀린다. **행마다 정합성이 깨진 상태**다.

## 5. 핵심 이상 징후

### F1. P2가 rowspan=2 컬럼을 잘못 처리

`bom_table_parser.py:166` 현재 코드:
```python
if h1_str and h2_str:
    merged_headers.append(f"{h1_str} | {h2_str}")
```

**문제**: 1행과 2행 텍스트가 같을 때(rowspan 복제 결과)도 그대로 `f"{h1} | {h2}"`로 이어붙인다.

**근본 원인**: P2 도입 시 입력 가정이 "1행 = 상위 그룹명 / 2행 = 보조 키워드"였고, **rowspan으로 양쪽에 동일 텍스트가 복제되는 케이스**를 분리해서 다루지 않았다.

### F2. 헤더 컬럼 수와 데이터 td 수의 정합성 미보장

- expand_table 결과 1행=16, 2행=15가 되어 P2의 `max_len` 처리에서 마지막 1열이 단독 잔여
- 일부 데이터 행은 td 7개만 있고, 헤더는 15~16개라 빈 셀이 앞에서부터 채워져 컬럼이 밀림
- `bom_table_parser.py` 또는 그 상위 단계에서 **각 행을 헤더 길이에 맞춰 패딩하는 정책**이 없음

### F3. P4 혼합문서 경고는 떴지만 추출 보강은 없음

`bom_pipeline.py:65~81`의 P4 로직은 `print` + `logger.warning`으로 끝난다.

- 사용자가 본 시나리오: BOM 프리셋으로 돌렸더니 BOM 1개만 나오고 경고 출력됨
- 사용자 후속 행동: `--preset estimate`로 **수동 재실행**
- 즉 경고는 떴지만 시스템이 자동으로 다음 단계로 가지 않음 → 진짜 페인포인트는 해소 안 됨

### F4. 두 번째 표(내역서) 통째로 누락

원본 md line 53에 14컬럼짜리 큰 내역서 표(`품 명/규격/단위/수량/재료비/노무비/경비/합계/비고`)가 존재한다.

- BOM 프리셋의 헤더 조건(`A=S/N|MARK|NO`, `B=SIZE|SPEC`, `C=Q'TY|WEIGHT`)은 이 표를 통과하지 못한다 (`품 명`이 A 그룹과 매칭 실패)
- 그래서 BOM 프리셋 단독 실행 시 내역서는 0개로 분류되어 빠진다
- P4 경고가 정확히 이 상황을 가리키지만, **경고에 그칠 뿐 자동 폴백/병행 실행은 미구현**

## 7. 원인 종합

### 원인 A. P2 입력 가정의 구멍

P2 설계 시 입력은 "1행 = 상위 그룹, 2행 = 보조 키워드"라는 단일 모델만 다뤘다.
실제 표는 **rowspan=2와 colspan=N이 섞인 하이브리드 구조**라서 expand 결과 1·2행에 같은 텍스트가 복제되는 경우가 흔하다.
이 케이스를 별도로 분기하지 않으면 `X | X` 형태의 무의미한 중복이 양산된다.

### 원인 B. 데이터 행 패딩 정책 부재

원본 OCR HTML이 빈 td를 누락하는 경우, 헤더 길이와 데이터 길이가 불일치한다.
이 시점에 **명시적인 정렬/패딩 보정**이 없으면 의미적 컬럼 매핑이 무너진다.
이번 케이스는 첫 행(td 7개) vs 두 번째 행(td 16개)처럼 행마다 들쭉날쭉한 입력에서 특히 치명적이다.

### 원인 C. P4의 책임 범위 한정

P4는 분석 보고서 §7.1 매트릭스에서 P6(자동 라우팅, 100줄+ 작업)과 분리해 가벼운 핫픽스로 도입됐다.
그래서 "운영자에게 알려준다"까지가 원래 책임이다.
하지만 **실제 사용자 워크플로**에서는 경고만으로는 부족하고 자동 재실행 또는 폴백이 필요한 상황이 드러났다.

## 8. 권장 조치

### 8.1 핫픽스 (즉시 착수)

#### P2-fix-1: rowspan 동일 텍스트 중복 제거

**위치**: `parsers/bom_table_parser.py:166`

```python
# Before
if h1_str and h2_str:
    merged_headers.append(f"{h1_str} | {h2_str}")
elif h2_str:
    merged_headers.append(h2_str)
else:
    merged_headers.append(h1_str)

# After
if h1_str and h2_str:
    if h1_str == h2_str:                     # rowspan=2 복제 케이스
        merged_headers.append(h1_str)
    else:
        merged_headers.append(f"{h1_str} | {h2_str}")
elif h2_str:
    merged_headers.append(h2_str)
else:
    merged_headers.append(h1_str)
```

**예상 영향**: `DESCRIPTION | DESCRIPTION` → `DESCRIPTION` 즉시 정상화. 회귀 위험 거의 없음.

**테스트 보강**: 기존 `test_two_row_header_is_merged_and_second_header_row_not_emitted_as_data`에 **rowspan 동일 텍스트 케이스 1건 추가**.

#### P2-fix-2: 1행/2행 길이 불일치 시 정책

`max_len` 기반 단순 padding 대신, **2행 길이가 1행보다 짧으면 부족분은 1행만 사용**하는 정책.
현재 `"비고" / "비고 | 비고"` 같은 16번째 컬럼 잔재가 사라진다.

### 8.2 단기 보강 (1~2일)

#### Bug B 보정: 데이터 행 패딩

`parse_html_tables` 또는 그 직후 단계에서 **각 데이터 행을 `len(headers)`에 맞춰 빈 문자열로 padding/truncate**.

단, 단순 끝쪽 패딩은 이번 케이스(앞쪽 td 누락)를 해결 못 한다. **`<td colspan="N">` 정보를 보존하면서 명시적 빈 셀을 채우는 expand 로직 검토 필요**. 이 부분은 expand_table 동작 정밀 분석 후 결정.

### 8.3 중기 (별도 Phase)

#### P4 격상 — 자동 폴백 또는 듀얼 실행

분석 보고서 §7.1의 P6에 해당. 100줄+ 변경, 회귀 위험 중간.

옵션 A — **자동 재실행 폴백**:
- BOM 프리셋 결과가 빈약하고(`total_tables ≤ 2`) 메타도 없으면, `--preset estimate`로 자동 재실행
- 두 결과 중 행 수가 더 많은 쪽 채택

옵션 B — **듀얼 실행 + 머지**:
- 두 프리셋 모두 돌리고 섹션을 합쳐서 출력
- 시트는 `BOM_자재표`, `견적서`, `내역서`가 모두 등장
- 사용자가 수동 재실행하던 흐름이 자동화됨

옵션 B가 사용자 시나리오에 더 부합하지만, 결과 머지 정책(중복 제거, 우선순위)이 추가로 필요.

| P-C | 데이터 행 패딩/얼라인 | `bom_table_parser.py` 상위 또는 expand_table | 30~80줄 | 중간 | 한 칸 밀림 해소 |
| P-D | P4 자동 폴백/듀얼 실행 | `bom_pipeline.py`, 진입점 | 100줄+ | 중간~높음 | 사용자 페인포인트 본질 해결 |

## 10. 최종 결론

이번 패치 자체가 잘못된 것은 아니다. **P1·P3은 의도대로 동작했고**, P2·P4도 단위 테스트 수준에서는 정확히 작동한다.

다만 두 가지 사실이 드러났다.

1. **단위 테스트의 입력 가정이 실제 PDF의 표 구조보다 단순했다.** rowspan/colspan 혼합이라는 실전 패턴을 회귀 셋에 추가해야 한다.
2. **P4는 알림에서 멈췄을 뿐, 사용자가 실제로 겪은 "수동 재실행" 페인을 해결하지 못했다.** 자동 폴백/듀얼 실행으로 격상이 필요하다.

한 줄 요약:

> **테스트가 통과해도 실측에서 깨질 수 있다. P2는 rowspan 케이스 한 줄 분기를 빠뜨렸고, P4는 경고만 띄우고 멈춰서 결국 사용자가 수동 재실행해야 했다.**

권장 다음 작업 순서:

1. P-A 핫픽스 (5줄 수정 + 회귀 테스트 1건)
2. P-B 길이 정합화
3. P-C 데이터 행 패딩 정책 정립 (expand_table 동작 분석 선행)
4. P-D 자동 폴백/듀얼 실행을 별도 Phase로 분리해 진행
