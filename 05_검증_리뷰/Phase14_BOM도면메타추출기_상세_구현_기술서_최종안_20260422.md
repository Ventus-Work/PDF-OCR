## 0. 문서 목적

Phase 12.5는 Document Pipeline의 JSON→Excel 렌더링 커버리지를 확장했지만, BOM Pipeline에는 별도의 빈 구간이 남아 있다. 현재 BOM 경로는 OCR 원문에 존재하는 도면 제목블록 메타데이터를 구조화 결과로 보존하지 않는다.

본 기술서는 Phase 14의 구현 범위를 다음과 같이 고정한다.

- `BomPipeline` 전용 문제만 다룬다.
- 메타 소스는 `BomExtractionResult.raw_text` 하나로 고정한다.
- v1은 **파일 단위 메타 1건**만 생성한다.
- 추가 OCR, bbox 재활용, 페이지별 메타 보장은 포함하지 않는다.
- 기존 BOM/LINE LIST 추출 결과는 유지하면서, 신규 `drawing_metadata` 필드와 `"도면_메타"` Excel 시트만 추가한다.

## 1. 현재 상태 실측

### 1.1 실제 손실 지점

현재 BOM 파이프라인의 핵심 흐름은 아래와 같다.

```text
OCR 엔진
  -> extract_bom_with_retry()
  -> extract_bom_tables() / extract_bom()
  -> BomExtractionResult(raw_text, bom_sections, line_list_sections)
  -> to_sections()
  -> JsonExporter / ExcelExporter
```

이 경로에서 도면 메타데이터가 사라지는 이유는 다음과 같다.

- `presets/bom.py`의 `blacklist`, `kill`, `noise_row`에는 `CLIENT:`, `PROJECT:`, `TITLE:`, `DRAWING NO`, `SCALE`, `DRW'D`, `CHK'D`, `APP'D` 등이 포함되어 있다.
- 즉 제목블록 메타 라인은 BOM 표 후보에서 제외되거나 활성 섹션을 종료시키는 역할만 하고, 구조화 결과에는 남지 않는다.
- 반면 `BomExtractionResult.raw_text`에는 OCR 원문 전체가 그대로 보존된다.

정리하면, **정답은 이미 `raw_text`에 있고 추출기만 비어 있다.**

### 1.2 현재 코드 기준 공백

| 파일 | 현재 상태 | 문제 |
|---|---|---|
| `ps-docparser/extractors/bom_types.py` | `raw_text`만 보존 | 메타 전용 필드 없음 |
| `ps-docparser/extractors/bom_ocr_retry.py` | OCR 재시도 후 `BomExtractionResult` 반환 | 메타 추출 주입점 없음 |
| `ps-docparser/extractors/bom_converter.py` | BOM/LL 섹션만 생성 | 메타 섹션 없음 |
| `ps-docparser/exporters/excel_builders.py` | 표/본문/주석/조건/교차참조/메타데이터 시트만 지원 | 도면 메타 시트 없음 |
| `ps-docparser/exporters/excel_exporter.py` | `tables`, `clean_text`, `notes`, `conditions`, `cross_references`, `revision_year`, `unit_basis`만 렌더 | `drawing_metadata` 미지원 |

### 1.3 핵심 결론

Phase 14 v1은 아래 원칙으로 해결한다.

- **state machine 수정 없음**
- **OCR 재호출 없음**
- **`raw_text` 정규식 파싱만 사용**
- **파일당 메타 dict 1건**

- 메타 단위: **파일 단위 1건**
- 소스: `BomExtractionResult.raw_text`

- 페이지별 메타 리스트
- 다중 도면 파일에서 도면별 메타 분리

## 3. 아키텍처 결정

### 3.1 전체 데이터 흐름

```text
PDF
  -> OCR
  -> extract_bom_with_retry()
     -> extract_bom_tables() / extract_bom()  # 기존 유지
     -> extract_drawing_meta(result.raw_text) # 신규
     -> BomExtractionResult(..., drawing_metadata={...})
  -> to_sections()
     -> DRAWING-META-1 섹션 prepend
     -> BOM-N / LL-N 섹션 기존대로 생성
  -> ExcelExporter
     -> 도면_메타 시트 조건부 생성
```

### 3.2 주입 위치 결정

메타 추출은 `extract_bom_with_retry()`의 최종 반환 직전에만 수행한다.

이 결정의 이유:

- `raw_text`가 가장 완성된 상태다.
- 1~4차 OCR 재시도 결과가 모두 반영된 뒤이므로 누락 가능성이 가장 낮다.
- `extract_bom()` / `extract_bom_tables()`를 순수 BOM 테이블 파서로 유지할 수 있다.

### 3.3 다중 매칭 처리

파일 단위 v1에서는 같은 필드에 대해 여러 값이 나올 수 있다. 이 경우 다음 규칙으로 고정한다.

- 위에서 아래로 line scan
- 각 필드는 **첫 번째 유효값만 채움**
- 한 번 채워진 필드는 이후 라인에서 덮어쓰지 않음

이 규칙은 구현자가 별도 의사결정을 하지 않도록 명시적으로 고정한다.

## 5. 추출 설계

### 5.1 전처리 규칙

`extract_drawing_meta()`는 아래 순서로 텍스트를 정리한다.

1. `raw_text`의 `<br>`, `<br/>`, `<br />`를 줄바꿈으로 치환
2. 잔여 HTML 태그 제거
3. `&nbsp;` 등 기본 HTML 엔티티를 평문으로 정규화
4. 줄 단위 split
5. 각 줄 `strip()` 및 연속 공백 축소

### 5.2 후보 라인 분해 규칙

메타는 같은 줄에 여러 필드가 함께 등장할 수 있으므로 아래 규칙을 고정한다.

- 줄에 `|`가 있고 메타 라벨이 하나라도 포함되면 `|` 기준으로 segment 분해 후 각 segment를 개별 후보로 검사
- 줄에 `|`가 있고 메타 라벨이 전혀 없으면 BOM 표 행으로 간주하고 스킵
- `---|---|---` 같은 구분선은 스킵

이 규칙으로 아래 형태를 동시에 처리한다.

```text
TITLE: PIPE SUPPORT DETAIL | DWG NO. KO-D-010-14-16N | REV. 0
```

### 5.3 필드 라벨 매핑

v1 라벨 매핑은 아래로 고정한다.

| 저장 필드 | 허용 라벨 |
|---|---|
| `dwg_no` | `DWG NO`, `DWG NO.`, `DRAWING NO`, `DRAWING NO.` |
| `rev` | `REV`, `REV.`, `REVISION` |
| `title` | `TITLE`, `TITLE.` |
| `date` | `DATE`, `ISSUED` |
| `project` | `PROJECT`, `JOB NO`, `JOB NO.` |
| `client` | `CLIENT`, `OWNER`, `CONTRACTOR` |
| `drawn_by` | `DRAWN BY`, `DRAWN`, `DRW'D` |
| `checked_by` | `CHECKED BY`, `CHECKED`, `CHK'D` |
| `approved_by` | `APPROVED BY`, `APPROVED`, `APP'D` |
| `scale` | `SCALE` |
| `sheet` | `SHEET` |

고정 매핑 규칙:

- `JOB NO`는 별도 필드로 만들지 않고 `project`에 저장
- `OWNER`, `CONTRACTOR`는 별도 필드로 만들지 않고 `client`에 저장
- 한글 값은 허용하지만, v1 라벨 앵커는 영문/약어만 지원

### 5.4 정규식 규칙

각 필드는 아래 원칙으로 검사한다.

- 라인 시작 anchored pattern 사용
- 구분자는 `:`, `.`, 공백을 허용
- 값은 segment 끝 또는 줄 끝까지 캡처
- 이미 값이 채워진 필드는 이후 매칭 무시

예시:

```python
DWG_NO:
  r"^(?:DWG\s*NO\.?|DRAWING\s*NO\.?)\s*[:.]?\s*(?P<value>.+?)\s*$"

REV:
  r"^(?:REV(?:ISION)?)\.?\s*[:.]?\s*(?P<value>.+?)\s*$"

TITLE:
  r"^TITLE\.?\s*[:.]?\s*(?P<value>.+?)\s*$"
```

### 5.5 값 정규화 규칙

값은 아래 규칙으로만 정리한다.

- 앞뒤 공백 제거
- 내부 연속 공백 2칸 이상을 1칸으로 축소
- 빈 문자열은 `None`
- 날짜, REV, SHEET 값의 포맷 변환 금지
- 대소문자 보정 금지

즉 OCR 원문을 최대한 유지하고, 표시 가능한 수준으로만 정리한다.

## 6. Excel 렌더링 설계

### 6.2 시트 레이아웃

시트 구조는 2열 고정으로 한다.

| 필드 | 값 |
| DWG NO | KO-D-010-14-16N |

필드 순서는 아래 고정 상수로 정의한다.

```python

    "title": "TITLE",
    "date": "DATE",

    "scale": "SCALE",
    "sheet": "SHEET",

```

### 6.3 Empty Guard

`ps-docparser/exporters/excel_exporter.py`에서는 아래 조건으로만 `"도면_메타"` 시트를 생성한다.

```python
has_any_drawing_meta = any(
    s.get("type") == "drawing_meta"
    and any(v is not None for v in s.get("drawing_metadata", {}).values())
    for s in sections
)
```

고정 규칙:

- 메타 섹션이 없으면 시트 생성 안 함
- 메타 dict가 있어도 값이 전부 `None`이면 시트 생성 안 함
- 메타 섹션은 계약상 1개만 존재한다고 가정하고 첫 번째 섹션만 사용

### 6.4 시트 순서

기존 Phase 12.5 순서를 유지하면서 `"도면_메타"`만 추가한다.

```text
견적서
내역서
조건
본문
주석
가감산_조건
교차참조
메타데이터
도면_메타
Table_N
```

BOM 모드에서는 일반적으로 `도면_메타`와 `Table_N`만 생성될 수 있다.

### 6.5 스타일

기존 스타일만 재사용한다.

- 헤더: `_FILL_HEADER`, `_FONT_HEADER`, `_ALIGN_CENTER`, `_BORDER_ALL`
- 본문: `_FONT_BODY`, `_ALIGN_LEFT`, `_BORDER_ALL`

컬럼 폭은 아래로 고정한다.

- A열 `필드`: `22`
- B열 `값`: `52`

## 9. 리스크와 대응

| 리스크 | 영향 | 대응 |
|---|---|---|
| 같은 파일 안에 서로 다른 도면 메타가 반복 | 중 | v1은 파일 단위 1건만 지원하며 첫 번째 유효값 우선으로 고정 |
| `TITLE: ... | DWG NO ...` 한 줄 다중 필드 | 중 | 메타 라벨 포함 pipe line은 segment 분해 |
| BOM 표 내부 값과 메타 라벨 오탐 | 상 | 라인 시작 anchored pattern + 라벨 없는 pipe row 스킵 |
| `DWG N0` 같은 OCR 오인식 | 중 | v1에서는 일반 오인식 보정 미포함, 테스트에 포함된 대표 변형만 대응 |
| HTML 태그가 포함된 OCR 출력 | 중 | `<br>` 치환 후 잔여 태그 제거 |
| Excel 빈 시트 생성 | 중 | exporter Empty Guard로 차단 |

## 11. 최종 고정 사양 요약

구현자가 더 이상 선택하지 않도록 마지막으로 핵심 결정을 다시 고정한다.

- `drawing_metadata`는 **dict 1건**
- 소스는 **`raw_text` only**
- 단위는 **파일 단위**
- 충돌 시 **첫 번째 유효값 우선**
- `DRAWING-META-1` 섹션 **prepend**
- `"도면_메타"` 시트 **조건부 생성**
- 추가 OCR / bbox / 페이지별 보장 **없음**
- 기존 BOM/LINE LIST 추출 동작 **불변**

이 문서를 Phase 14의 저장용 최종 기술서로 사용한다.
