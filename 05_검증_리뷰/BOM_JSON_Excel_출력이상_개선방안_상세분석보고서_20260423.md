## 1. 목적

기존 보고서는 “왜 결과가 기대와 다르게 나왔는가”를 정리했다.  
본 보고서는 그 후속으로, 이미 제시된 개선 아이디어를 **실제 코드 구조 기준으로 재검토**하고, 구현 우선순위와 리스크를 포함한 **실행 가능한 개선안**으로 정리한다.

핵심 질문은 아래 5가지다.

1. 복합 헤더 문제는 어디를 고쳐야 가장 효과적인가
2. JSON 단계의 컬럼 손실은 어디서 막아야 하는가
3. Excel 시트명을 더 의미 있게 만들려면 어느 계층을 손봐야 하는가
4. 견적/내역/자재표 혼합 문서 라우팅은 어느 Phase로 보내는 것이 맞는가
5. Phase 14 메타 추출은 어디까지 확장하는 것이 안전한가

---

## 2. Executive Summary

제안된 개선안의 방향성은 전반적으로 맞다. 다만 코드 구조를 기준으로 보면 일부는 **즉시 적용 가능한 저위험 수정**, 일부는 **추가 설계가 필요한 중위험 변경**, 일부는 **별도 Phase 급 구조 변경**으로 나뉜다.

최종 판단은 아래와 같다.

- **즉시 착수 권장**
  - `2순위`: 중복 헤더 dedupe
  - `1순위`: 다단 헤더 병합
- **보완 설계 후 착수 권장**
  - `3순위`: Excel 시트명/분류 개선
  - `5순위`: 메타 추출 fallback 확장
- **별도 Phase 분리 권장**
  - `4순위`: 문서 종류 자동 판정 + 듀얼 파이프라인 머지

가장 중요한 해석은 다음이다.

> 이번 문제의 본체는 먼저 **복합 헤더 파싱 + 중복 헤더 overwrite**이고,  
> 그 다음이 **출력 표현(시트명)**이며,  
> 문서 자동 라우팅과 메타 추출 확장은 그보다 한 단계 뒤 작업이다.

## 3. 제안안 상세 검토

### 3.1 1순위 제안: 복합 헤더 파서 보강

#### 제안 요약

`bom_table_parser.py`에서 `effective_grid[0]`만 헤더로 쓰지 말고, 다행 헤더를 감지해 병합하자는 제안이다.

기준 코드:

- `ps-docparser/parsers/bom_table_parser.py:140`
- `ps-docparser/parsers/bom_table_parser.py:141`

현재 구현:

```python
headers = effective_grid[0]
rows = effective_grid[1:]
```

#### 왜 필요한가

이번 문제 문서는 아래 구조를 가진다.

- 1행 헤더: `DESCRIPTION`, `DWG NO.`, `MAT'L`, `SIZE`, `자재중량 [Kg]`, `자재면적 [m2]` ...
- 2행 헤더: `UNIT`, `WEIGHT`, `LOSS`, `WEIGHT`, `UNIT`, `m2`, `LOSS`, `m2`

현재 로직에서는 2행 헤더가 데이터로 떨어지므로:

- JSON 1행이 가짜 데이터가 되고
- Excel 2행이 실제 데이터가 아니라 헤더 반복이 되며
- 이후 dedupe를 해도 “이미 잘못된 헤더 구조” 위에서 동작하게 된다

#### 제안안 평가

방향은 맞다. 다만 아래 보완이 필요하다.

- `_detect_header_depth()`는 단순히 `UNIT/WEIGHT/LOSS/m2` 키워드만 보지 말고, **숫자 비율이 낮은지**도 같이 봐야 한다.
- 이유: 어떤 데이터 첫 행도 전부 문자열일 수 있어 오판 가능성이 있다.
- `max_depth=3`은 적당하다.
- 이번 케이스 기준으로는 `depth=2`가 가장 자연스럽다.

#### 권장 구현 원칙

헤더 깊이 판단은 아래 3조건을 함께 보는 것을 권장한다.

1. 현재 행에 숫자형 셀이 거의 없음
2. `UNIT`, `WEIGHT`, `LOSS`, `m2`, `kg`, `wt`, `합계`, `소계` 같은 보조 헤더 토큰이 존재
3. 이전 행이 colspan 전개로 동일값 반복 패턴을 가짐

#### 권장 결론

**채택 권장.**

다만 “고정 keyword만 보면 된다” 수준이 아니라,  
`보조 헤더 토큰 + 숫자 비율 + 반복 텍스트 패턴`을 함께 보는 휴리스틱으로 구현하는 것이 안전하다.

### 3.2 2순위 제안: 중복 헤더명 dict overwrite 방지

#### 제안 요약

`bom_converter.py`에서 row dict 생성 전에 헤더를 dedupe해서, 동일 컬럼명이 JSON에서 덮어써지지 않도록 하자는 제안이다.

기준 코드:

- `ps-docparser/extractors/bom_converter.py:61`
- `ps-docparser/extractors/bom_converter.py:95`

현재 구현:

```python
row_dict[key] = cell
```

#### 왜 필요한가

이번 테이블은 `자재중량 [Kg]`가 4개, `자재면적 [m2]`가 4개 반복된다.  
dict는 동일 key를 1개만 유지하므로 뒤 값이 앞 값을 덮어쓴다.

즉 현재 손실은 Excel이 아니라 **JSON 생성 시점**에 이미 발생한다.

#### 제안안 평가

이 제안은 매우 적절하다.

장점:

- blast radius가 작다
- 즉시 JSON 손실을 멈출 수 있다
- 현재 파서 구조를 거의 건드리지 않아도 된다

주의점:

- 단순히 `"열"` 같은 fallback 이름을 쓰면 새 중복이 생긴다
- 빈 헤더는 `열7`, `열8`처럼 **열 번호 기반**으로 만들어야 한다
- dedupe는 `tables[].headers`와 `row_dict` 양쪽에 동시에 적용해야 한다

#### 권장 구현 원칙

아래와 같이 고정하는 것이 좋다.

```python
def _dedupe_headers(headers):
    seen = {}
    out = []
    for idx, h in enumerate(headers, start=1):
        base = str(h).strip() or f"열{idx}"
        n = seen.get(base, 0) + 1
        seen[base] = n
        out.append(base if n == 1 else f"{base}_{n}")
    return out
```

#### 권장 결론

**최우선 착수 권장.**

이번 케이스처럼 컬럼 손실이 눈에 보이는 경우, 이 수정은 적은 코드 변경으로 가장 큰 효과를 낸다.

### 3.3 1순위와 2순위의 관계

이 두 작업은 독립이 아니라 **직렬 의존** 관계다.

- 2순위만 먼저 해도 overwrite는 막을 수 있다
- 하지만 1순위가 없으면 `WEIGHT`, `m2` 보조 헤더 행이 여전히 데이터로 남는다
- 1순위만 먼저 하면 헤더 구조는 좋아지지만 중복 key overwrite는 여전히 남는다

따라서 실제 구현 전략은 아래가 가장 좋다.

1. `dedupe`를 먼저 넣어 즉시 손실 중단
2. 이어서 `multi-header merge`를 넣어 구조 복원
3. 두 변경을 함께 검증

즉 **핫픽스는 2순위, 구조적 정상화는 1순위**로 이해하는 것이 맞다.

### 3.4 3순위 제안: 견적/내역/조건 시트 자동 분기 또는 BOM 시트 의미화

#### 제안 요약

`excel_classifier.py`에 BOM 자재표 인식을 추가해서, 지금처럼 `"Table"`만 나오지 않게 하자는 제안이다.

#### 현재 코드 구조상 주의점

여기에는 중요한 구조 제약이 있다.

현재 `_classify_table()`이 반환하는 값은 아래 4개뿐이다.

- `estimate`
- `detail`
- `condition`
- `generic`

그리고 `excel_exporter.py`도 이 네 가지 분기만 처리한다.

즉 `_classify_table()`에서 새로 `return "bom"`을 추가하면,  
**exporter 쪽에서도 그 분기를 받아야 한다.**

그렇지 않으면 새 타입이 생겨도 결과는 여전히 generic 처리되거나 누락될 수 있다.

#### 제안안 평가

아이디어 자체는 맞지만, 구현 방법은 두 갈래가 있다.

##### 선택지 A. 낮은 리스크

`_classify_table()`는 그대로 두고, generic 시트 생성 시 아래 조건이면 시트명을 `BOM_자재`로 바꾼다.

- `tbl["type"] == "BOM_자재"`
- 또는 헤더에 `DWG NO.`, `MAT'L`, `WEIGHT` 계열이 3개 이상

장점:

- exporter 구조 변경이 작다
- 기존 `generic` 빌더 재사용 가능

##### 선택지 B. 더 큰 구조 변경

- classifier에 `bom` 타입 추가
- exporter에 `bom_tables` 큐 추가
- `"BOM_자재"` 또는 `"LINE_LIST"` 시트를 별도 생성

장점:

- 의미가 분명해진다

단점:

- 구현 범위가 넓어진다
- 기존 generic 흐름과 테스트를 더 건드린다

#### 권장 결론

이번 단계에서는 **선택지 A 권장**.

즉 “새 분류 타입 추가”보다,  
**generic 결과를 더 알아보기 쉬운 이름으로 출력하는 방식이 저위험 고효율**이다.

---

### 3.5 4순위 제안: 문서 종류 자동 판정 + 프리셋 라우팅

#### 제안 요약

문서 시작 전에 키워드 빈도를 스캔해,

- 견적/내역서 성격이 강하면 DocumentPipeline
- BOM 도면 성격이 강하면 BomPipeline
- 둘 다 강하면 이중 실행 후 머지

로 가자는 아이디어다.

#### 왜 바로 하기 어려운가

이건 단순 parser patch가 아니라, 아래를 함께 건드리는 구조 변경이다.

- 파이프라인 진입점
- 프리셋 선택 정책
- 결과 병합 정책
- 파일명/출력 경로 정책
- 중복 데이터 충돌 해결

특히 “이중 실행 후 결과 머지”는 설계 결정이 많다.

- JSON 머지 단위는 섹션인가 문서인가
- Excel 머지는 시트명 충돌 시 어떻게 처리하는가
- 동일 정보가 두 파이프라인에 모두 있으면 어느 쪽을 우선하는가

#### 권장 결론

**이번 수정 범위에 넣지 않는 것이 맞다.**

이 작업은 parser hotfix가 아니라 **별도 Phase**로 다뤄야 한다.

다만 바로 넣을 수 있는 작은 개선은 있다.

- `preset=bom`인데 문서 초반에 `견적서`, `내역서`, `일반사항`, `금액`, `품명`이 반복되면 warning 로그 출력
- 즉 자동 라우팅 대신 **사전 경고**부터 넣는 방식

이 정도는 저위험으로 넣을 수 있다.

### 3.6 5순위 제안: Phase 14 메타 추출 범위 확장

#### 제안 요약

현재 `drawing_meta.py`는 line-anchored title block 패턴만 본다.  
이번 문서처럼 title block 라인이 없으면 결과가 전부 `None`이 되는 것이 정상이다.

제안된 옵션:

- 옵션 A: 현행 유지 + 사용자 안내 로그
- 옵션 B: `DWG NO.` 컬럼 값을 fallback으로 사용

#### 제안안 평가

##### 옵션 A

장점:

- 구현이 쉽다
- 정책 오해를 줄인다
- 잘못된 메타를 넣을 가능성이 거의 없다

단점:

- 이번 문서에서는 여전히 메타 시트가 비게 된다

##### 옵션 B

장점:

- 일부 문서에서 `dwg_no`만이라도 채울 수 있다

단점:

- 표 내부 `DWG NO.`는 “도면 메타”가 아니라 **행별 참조값**일 수 있다
- 여러 도면 번호가 섞인 경우 잘못된 대표값이 된다

#### 더 안전한 절충안

옵션 B를 쓰려면 최소한 아래 조건이 필요하다.

- `DWG NO.` 컬럼의 non-empty unique 값이 **정확히 1개일 때만**
- 그때만 `drawing_metadata["dwg_no"]` fallback 허용
- 그리고 provenance를 로그로 남김
  - 예: `dwg_no source = table_fallback`

#### 권장 결론

현재 단계에서는 **옵션 A 즉시 적용**,  
옵션 B는 **정책 합의 후 제한적 fallback**으로 분리하는 것이 가장 안전하다.

## 4. 권장 작업 순서 재정의

기존 제안의 순서를 코드 리스크 기준으로 재정렬하면 아래가 가장 좋다.

### Step 0. 재현 fixture 고정

먼저 이번 문제 문서를 테스트 fixture로 고정한다.

이 단계가 먼저 필요한 이유:

- 수정 후 “정말 나아졌는지”를 객관적으로 비교해야 함
- 특히 JSON 첫 행의 가짜 헤더, duplicate overwrite, 시트명 변화를 snapshot으로 잡을 수 있음

### Step 1. `_dedupe_headers()` 도입

목표:

- JSON 단계 손실 즉시 중단

완료 기준:

- `자재중량 [Kg]`, `자재중량 [Kg]_2`, `..._3`, `..._4` 생성
- 동일 key overwrite 없음

### Step 2. 다단 헤더 병합

목표:

- `WEIGHT`, `m2` 보조 헤더가 데이터 행으로 들어가는 문제 제거

완료 기준:

- JSON 첫 행이 실제 자재 데이터부터 시작
- merged header가 `자재중량 [Kg] / WEIGHT` 같은 구조로 생성

### Step 3. Excel 시트명 의미화

목표:

- `"Table"` 대신 `"BOM_자재"` 또는 유사한 의미 있는 이름 제공

완료 기준:

- 사용자가 workbook만 열어도 이 시트가 BOM 자재표인지 이해 가능

### Step 4. 경고 로그 추가

목표:

- BOM preset이 비순수 BOM 문서를 받았을 때 사용자 혼동 완화

완료 기준:

- `견적/내역/조건` 키워드가 강하면 warning 출력
- title block 메타 미검출 시 안내 로그 출력

### Step 5. 정책성 기능 분리

대상:

- 문서 자동 라우팅
- table 기반 dwg_no fallback

이 단계는 별도 기술서로 분리하는 것이 맞다.

## 5. 테스트 계획

### 5.1 파서 레벨

신규 테스트 권장:

- 2행 헤더 HTML 표 입력 -> merged header 생성 확인
- duplicate header 입력 -> `_dedupe_headers()` 결과 확인
- 단일 헤더 기존 BOM 표 -> 무회귀 확인

### 5.2 JSON 변환 레벨

검증 포인트:

- JSON 첫 데이터 행이 더 이상 `WEIGHT`, `m2` 헤더 반복이 아님
- duplicated key overwrite 없음
- `tables[0].headers`와 `rows[0].keys()`가 일관됨

### 5.3 Excel 레벨

검증 포인트:

- 시트명 `"Table"` -> `"BOM_자재"` 또는 합의된 이름
- 첫 데이터 행이 실제 자재 행
- 열 수가 JSON과 일치

### 5.4 메타 추출 레벨

검증 포인트:

- 현재 문서에서는 title block 메타가 없으므로 전부 `None`
- 대신 안내 로그가 뜨는지 확인
- 제한적 fallback을 도입할 경우 unique 값 1개일 때만 동작하는지 확인

## 6. 최종 판단

이번 개선안 중 실제 효과가 가장 큰 것은 아래 두 가지다.

1. **복합 헤더 병합**
2. **중복 헤더 dedupe**

이 두 작업이 끝나면, 현재 보고서에서 지적한 “JSON/XLSX가 깨진 느낌”의 대부분이 해결된다.

반면 아래 두 가지는 중요하지만 즉시 핵심은 아니다.

- 문서 자동 라우팅
- 메타 추출 범위 확장

이 둘은 정책과 제품 방향까지 얽혀 있으므로, hotfix와 같은 묶음으로 처리하면 오히려 리스크가 커진다.

최종적으로 추천하는 작업 묶음은 다음과 같다.

### 이번 턴/이번 PR 권장 범위

- `_dedupe_headers()` 도입
- 다단 헤더 감지/병합
- generic BOM 시트명 의미화
- 경고 로그 추가
- 회귀 테스트 추가

### 다음 Phase로 넘길 범위

- 문서 자동 판정
- 듀얼 파이프라인 동시 실행 및 머지
- table 기반 `dwg_no` fallback

---

## 7. 한 줄 결론

> **지금 가장 먼저 고쳐야 할 것은 “메타 추출기”가 아니라, “복합 헤더 파서 + 중복 헤더 overwrite”다.**

그 다음이 사용자 인지성을 위한 시트명/경고 로그이고,  
자동 라우팅과 메타 fallback은 별도 정책 작업으로 분리하는 것이 맞다.
