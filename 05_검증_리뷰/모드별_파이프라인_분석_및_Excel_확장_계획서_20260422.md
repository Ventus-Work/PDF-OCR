## 1. 사용자 가설 요약

> "MD 파일에는 전체 내용이 있는데 JSON·Excel은 BOM 표만 나옵니다.
> `bom_table_parser.py`·`bom_state_machine.py`가 BOM 키워드로만 행을 필터링하기 때문이고,
> 다른 내용도 Excel에 넣으려면 Extractor에 키워드를 추가해야 합니다."

### 1.1 가설의 정확성 평가

| 가설 진술 | 판정 | 비고 |
|---|---|---|
| MD는 전체 OCR 원문을 보존 | ✅ 정확 | 두 파이프라인 모두 MD는 손실 없이 저장 |
| BOM state machine이 앵커·헤더·킬 키워드 기반 필터 | ✅ 정확 | `extractors/bom_extractor.py` `extract_bom()` 상태머신 |
| JSON·Excel에는 필터링된 BOM만 들어감 | ⚠️ **BOM 모드에서만 참** | Document 모드는 반대로 JSON이 매우 풍부 |
| Extractor에 키워드 추가가 해법 | ❌ **오진** | 실제 병목은 Excel Exporter 렌더링 커버리지 |

### 1.2 핵심 오해

사용자 가설은 **"단일 파이프라인 + 단일 필터"** 모델을 가정했으나, 실제 코드는 **듀얼 파이프라인**(`BomPipeline` / `DocumentPipeline`)이 `pipelines/factory.py`에서 프리셋으로 분기된다. 두 파이프라인은 필터 위치·단위·손실 지점이 모두 다르다.

## 2. 실측 결과 — 듀얼 파이프라인 아키텍처

### 2.1 파이프라인 분기 (이미 구현됨)

```
                      ┌─── BomPipeline (preset=bom) ─────────────────┐
                      │   OCR → extract_bom_with_retry (4차 재시도)  │
                      │   → state_machine 행 필터                    │
                      │   → to_sections()                             │
                      │       clean_text="", notes=[], conditions=[], │
                      │       cross_references=[], tables=[BOM 행]    │
                      │   → JSON: BOM 행만                            │
                      │   → Excel: BOM 행만 (Table_N 시트)            │
                      └───────────────────────────────────────────────┘

                      ┌─── DocumentPipeline (preset=pumsem/estimate/None) ─┐
                      │   PDF → hybrid_extractor → MD                       │
                      │   MD → parse_markdown()                             │
                      │     1. split_sections() — TOC 마커 기반             │
                      │     2. process_section_tables() — HTML 표 파싱      │
                      │     3. process_section_text() — 본문/주석/조건 추출 │
                      │   → JSON: 전체 필드 보존 (clean_text/notes/         │
                      │           conditions/cross_references/              │
                      │           revision_year/unit_basis/tables)          │
                      │   → Excel: **tables만 렌더링** ← 실제 손실 지점!    │
                      └─────────────────────────────────────────────────────┘
```

### 2.2 JSON vs Excel 데이터 커버리지 비교표

| JSON 필드 | BOM 모드 | Document 모드 JSON | Document 모드 Excel |
|---|---|---|---|
| `section_id` | 있음 | 있음 | (시트 이름용) |
| `title` | 있음 | 있음 | 표지/시트명 |
| `chapter`, `department` | 없음 | 있음 | ❌ 미렌더 |
| `page` | 있음 | 있음 | ❌ 미렌더 |
| `clean_text` (본문) | **""** | 있음 | ❌ **미렌더** |
| `tables` | BOM 행 | 있음 | ✅ 4종 시트 분기 |
| `notes` (주석) | **[]** | 있음 | ❌ **미렌더** |
| `conditions` (가감산) | **[]** | 있음 | ❌ **미렌더** |
| `cross_references` (교차참조) | **[]** | 있음 | ❌ **미렌더** |
| `revision_year` | **None** | 있음 ("" 또는 값) | ❌ **미렌더** |
| `unit_basis` | **None** | 있음 ("" 또는 값) | ❌ **미렌더** |

### 2.3 증거 — 핵심 코드 인용

**BomPipeline의 sections 생성** (`extractors/bom_extractor.py:513-591` `to_sections()`):
```python
sections.append({
    "section_id": f"BOM-{i}",
    "clean_text": "",            # ← BOM 모드는 본문 강제 공백
    "notes": [],                 # ← 강제 빈 배열
    "conditions": [],
    "cross_references": [],
    "revision_year": None,
    "unit_basis": None,
    "tables": [{...BOM 행...}],
})
```

**DocumentPipeline의 sections 생성** (`parsers/text_cleaner.py:324-339` `process_section_text()`):
```python
return {
    "section_id": section["section_id"],
    "clean_text": clean,             # ← 전체 본문 보존
    "tables": section.get("tables", []),
    "notes": notes,                  # ← 추출된 [주] 블록
    "conditions": conditions,        # ← {type, condition, rate} dict 리스트
    "cross_references": cross_references,
    "revision_year": revision_year,
    "unit_basis": unit_basis,
}
```

**Excel Exporter의 손실 지점** (`exporters/excel_exporter.py:591-608` `_export_impl()`):
```python
for section in sections:
    # clean_text는 오직 문서 제목 첫 줄로만 사용
    if not doc_title:
        clean = section.get("clean_text", "")
        doc_title = clean.split("\n")[0].strip()[:60] if clean else ""

    # 이후는 tables만 순회 — notes/conditions/cross_references/revision_year
    # /unit_basis는 절대 Excel에 기록되지 않음
    for tbl in section.get("tables", []):
        kind = _classify_table(tbl)
        if kind == "estimate": ...
        elif kind == "detail": ...
        elif kind == "condition": ...   # 이것은 헤더에 "일반사항/특기사항"이 든 '표'일 뿐,
                                        # JSON의 conditions[]와는 다른 데이터
        elif kind == "generic": ...
```

**→ 결론**: Document 모드에서 **JSON은 이미 풍부**하다. Excel 렌더러만 `tables` 키 외 다른 필드를 무시하고 있다.

## 3. 문제 재정의 — 두 개의 독립 이슈

사용자 문의를 하나의 "필터 강도 조절" 문제로 보면 오진이 된다. 두 개의 독립 문제로 분리해야 한다.

### 3.1 Problem A — BOM 모드 필터 폭 (선택적 확장)

| 항목 | 내용 |
|---|---|
| 현상 | MD에는 있는 도면 메타(DWG NO, REV, TITLE, DATE)·주변 설명이 JSON/Excel에서 사라짐 |
| 원인 | `extract_bom()` state machine이 BOM/LL 앵커-헤더 패턴에만 반응, 나머지는 drop |
| 아키텍처 위치 | **Extractor 레벨** (state_machine/converter) |
| 영향 범위 | BOM 파이프라인만 |
| 긴급도 | 중 (사용자 수동 재작업 유발) |

### 3.2 Problem B — Document 모드 Excel 렌더링 커버리지 (실제 병목)

| 항목 | 내용 |
|---|---|
| 현상 | JSON에 있는 본문·주석·조건·참조·메타가 Excel에 안 보임 |
| 원인 | `_export_impl()`이 `section.get("tables", [])`만 순회 |
| 아키텍처 위치 | **Exporter 레벨** (excel_builders) |
| 영향 범위 | Document 파이프라인만 (pumsem/estimate/None) |
| 긴급도 | **상** (JSON에 데이터 있음에도 Excel에 없음 → 사용자 체감 가장 큼) |

### 3.3 두 문제의 비대칭성

| 축 | Problem A | Problem B |
|---|---|---|
| 데이터 유무 | JSON에도 없음 | **JSON에는 있음** |
| 해법 난이도 | 중 (키워드·bbox 설계 필요) | 하 (기존 JSON 필드 렌더링만) |
| 회귀 위험 | 중 (state_machine 변경) | 낮음 (빌더 추가) |
| 테스트 부담 | OCR smoke 필요 | 단위 테스트로 충분 |
| 사용자 체감 가치 | 중 | **상** |

**→ Problem B를 먼저, Problem A는 별도 Phase로 분리**하는 것이 합리적이다.

### 4.1 Problem A 해법 (BOM 모드)

| 코드 | 해법 | 영향 모듈 | 난이도 | 우선순위 |

### 4.2 Problem B 해법 (Document 모드 Excel)

| 코드 | 해법 | 영향 모듈 | 난이도 | 우선순위 |
|---|---|---|---|---|

| **B5** | **메타 시트** (`_build_meta_sheet`) | `excel_builders.py` | 하 | 중 |

### 4.3 해법 선택 근거

- **B1~B5가 최우선**: JSON에 데이터가 있으므로 렌더러 추가만으로 해결. Blast radius 최소, 회귀 위험 낮음.
- **Problem A는 Problem B 이후**: BOM 모드 사용자는 현재 도면 메타를 수동 재작업 중일 가능성이 있으나, 데이터 자체가 JSON에 없으므로 OCR·키워드 설계가 선행되어야 함.
- **순서상 Phase 12 분해 완료 후 B1~B5 진행이 유리**: Phase 12 Step 12-2로 `excel_builders.py` 모듈이 분리된 뒤 시트 빌더를 추가하면 국소 변경으로 끝남. Phase 12 이전에 진행하면 720줄 `excel_exporter.py`를 더 키우게 됨.

## 6. Phase 14 (Problem A) 예비 스케치

Problem A는 본 계획서의 **비범위**이지만, 향후 Phase 14 착수 시 참고 가능한 예비 아이디어만 기록한다.

### 6.1 추출 대상 후보

- DWG NO (도면 번호)
- REV / REVISION (개정 번호)
- TITLE (도면 제목)
- DATE / ISSUED (발행일)
- PROJECT / JOB NO
- CLIENT / OWNER
- DRAWN BY / CHECKED BY / APPROVED BY
- SCALE / SHEET

### 6.2 접근 방식 후보

1. **키워드 앵커 + 고정 패턴**: ZAI/Mistral OCR이 타이틀 블록을 파이프 구분 텍스트로 내놓는 경우가 많음. 앵커 키워드(`DWG`, `REV`, `DATE`) 뒤 고정 거리 내 값 추출.
2. **bbox 기반 영역 분리**: 타이틀 블록 bbox 내부에서 셀 단위로 키-값 쌍 매칭.
3. **2차 OCR 프롬프트 특화**: 타이틀 블록만 크롭한 이미지에 "이 도면의 메타데이터를 JSON으로 출력하라" 프롬프트 투입.

### 6.3 출력 구조 후보

`BomExtractionResult`에 `drawing_metadata: dict` 필드 추가, `to_sections()`에서 별도 섹션(`section_id = "DRAWING-META-{i}"`)으로 내보내 Excel에서 기존 테이블 시트와 동일 경로로 렌더링.

상세는 Phase 14 착수 시 별도 기술서로 작성한다.
