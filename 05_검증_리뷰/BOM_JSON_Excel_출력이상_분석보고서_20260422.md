## 1. Executive Summary

이번 결과는 **파일 생성 실패**가 아니라, **입력 문서 성격과 현재 BOM 파이프라인의 해석 방식이 어긋나 기대한 내용이 나오지 않은 케이스**로 판단된다.

핵심 결론은 아래 4가지다.

1. 테스트 대상 문서는 전형적인 `BOM 도면(title block + BOM/LINE LIST)`이 아니라, **견적서 + 내역서 + 자재표가 함께 있는 혼합 문서**에 가깝다.
2. 그래서 BOM 프리셋은 문서 전체를 풍부하게 구조화하지 않고, **맨 뒤쪽의 BOM 유사 테이블 1개만** JSON/XLSX로 내보냈다.
3. 마지막 자재표는 **복합 헤더(2행 헤더 + 중복 컬럼명)** 구조인데, 현재 BOM HTML 파서는 이를 제대로 보존하지 못한다.
4. 이 손실은 JSON 단계에서 끝나지 않는다. **JSON -> Excel 직렬 파이프라인** 구조이므로, JSON에서 잘못 만들어진 컬럼 구조와 overwrite 손실이 Excel로 그대로 전파된다.

중요한 해석 기준은 아래와 같다.

- 본 보고서에서 `정상`은 **현재 코드가 그렇게 동작하도록 짜여 있고 실제로 그 경로대로 실행됐다**는 뜻이다.
- 이것은 **사용자 기대에 부합한다**거나 **수정이 불필요하다**는 뜻이 아니다.
- 즉 `코드 동작상 예상 가능`과 `제품 관점에서 만족스러움`은 구분해서 읽어야 한다.

한 줄 요약:

> **Phase 14가 죽은 것이 아니라, 입력 문서가 BOM 전제와 다르고 마지막 표가 복합 헤더라서 JSON/XLSX가 함께 깨진 것이다.**

## 3. 산출물 실측 결과

### 3.1 MD 원문 관찰

`..._bom.md`에는 아래 성격의 내용이 함께 존재한다.

- 상단 견적서 본문
- 일반사항 / 특기사항
- `## 내 역 서`
- 상세 내역 HTML 테이블
- 마지막에 `DESCRIPTION | DWG NO. | MAT'L | SIZE ...` 형태의 자재표

즉 이 문서는 순수 BOM 도면이라기보다, **견적 문서에 BOM 유사 테이블이 포함된 하이브리드 문서**다.

### 3.2 JSON 실측

`..._bom.json`은 최종적으로 **`BOM-1` 섹션 1개만** 생성했다.

관찰된 특징:

- `DRAWING-META-1` 섹션 없음
- `clean_text`, `notes`, `conditions`, `cross_references` 모두 비어 있음
- 마지막 자재표만 `tables[0]`로 존재
- 첫 번째 데이터 행이 사실상 **두 번째 헤더 행**이다
  - 예: `DESCRIPTION`, `DWG NO.`, `MAT'L`, `SIZE`, `WEIGHT`, `m2`
- 동일 헤더명이 반복되어 일부 컬럼 값이 덮어써진다
  - 예: `자재중량 [Kg]` 4개, `자재면적 [m2]` 4개

### 3.3 Excel 실측

`..._bom.xlsx`에는 시트가 **`Table` 1개만** 생성되었다.

상위 8행 관찰 결과:

- 1행은 헤더
- 2행은 실제 데이터가 아니라 **보조 헤더(WEIGHT / m2)**
- 이후부터 자재 행이 이어짐
- `"도면_메타"` 시트 없음
- `"견적서"`, `"내역서"`, `"조건"` 시트도 없음

즉 Excel도 JSON과 동일하게 **범용 테이블 1장만 보존된 상태**다.

## 4. 핵심 이상 징후

### F1. 기대한 문서 전체 구조가 아니라 BOM 유사 테이블 1개만 추출됨

현재 BOM 프리셋의 헤더 감지 조건은 `A ∧ B ∧ C` 그룹 동시 만족이다.

- `bom_header_a`: `S/N`, `SN`, `MARK`, `NO`, `NO.`
- `bom_header_b`: `SIZE`, `SPEC`, `SPECIFICATION`
- `bom_header_c`: `Q'TY`, `QTY`, `QUANTITY`, `WT`, `WEIGHT`, `WT(KG)`
- 근거: `ps-docparser/presets/bom.py:24-26`

이번 문서의 상단 견적서/내역서/조건 표는 이 조건을 충족하지 않는다. 반면 마지막 자재표는 `DWG NO.`, `SIZE`, `WEIGHT` 계열을 포함해 BOM 유사 표로 판정될 가능성이 높다.

따라서 현재 결과는 “코드가 오작동했다”기보다, **BOM 프리셋이 이 문서를 그렇게 해석하도록 설계된 결과**에 가깝다. 다만 이것은 코드 동작 설명일 뿐이고, **사용자 기대와 맞지 않는다면 제품 관점에서는 수정 대상**이다.

### F2. Phase 14 메타 추출기는 동작했지만 결과가 전부 `None`

현재 코드에는 Phase 14 메타 추출 주입 훅이 이미 들어 있다.

- `result.drawing_metadata = extract_drawing_meta(full_text)`
- 근거: `ps-docparser/extractors/bom_ocr_retry.py:133`

또한 `to_sections()`도 메타가 있으면 `DRAWING-META-1`을 prepend하도록 되어 있다.

- 근거: `ps-docparser/extractors/bom_converter.py:34`, `:50`

하지만 실제 `..._bom.md` 원문에 대해 `extract_drawing_meta()`를 직접 적용한 결과는 아래와 같았다.

```python
{
  'dwg_no': None,
  'rev': None,
  'title': None,
  'date': None,
  'project': None,
  'client': None,
  'drawn_by': None,
  'checked_by': None,
  'approved_by': None,
  'scale': None,
  'sheet': None
}
```

원인:

- 현재 메타 추출기는 **라인 시작 anchored pattern**만 본다.
- 근거: `ps-docparser/extractors/drawing_meta.py:183`, `:239-250`
- 그런데 이번 MD에는 `TITLE: ...`, `PROJECT: ...`, `REV. 0`, `SCALE: ...` 같은 **타이틀 블록형 메타 라인**이 보이지 않는다.
- `DWG NO.`는 존재하지만 **컬럼 헤더**로만 존재하고, `DWG NO. KO-...` 형태의 메타 라인이 아니다.

결론적으로, `"도면_메타"` 시트가 안 나온 이유는 **기능 미호출이 아니라, 입력 원문에 현재 정규식이 잡을 메타 라인이 없기 때문**이다.

### F3. 복합 헤더가 데이터 행으로 잘못 들어감

현재 BOM HTML 파서는 `effective_grid[0]`을 헤더, `effective_grid[1:]`를 데이터로 고정한다.

- 근거: `ps-docparser/parsers/bom_table_parser.py:140-141`

문제는 이번 자재표가 아래 구조라는 점이다.

- 헤더 1행: `DESCRIPTION | DWG NO. | MAT'L | SIZE | ... | 자재중량 [Kg] ...`
- 헤더 2행: `UNIT | WEIGHT | LOSS | WEIGHT | UNIT | m2 | LOSS | m2`

즉 **2행 헤더 구조**인데, 현재 파서는 첫 행만 헤더로 쓰고 둘째 헤더 행을 데이터로 밀어 넣는다.

그래서 JSON/XLSX 2행에 `WEIGHT`, `m2`가 데이터처럼 들어갔다.

### F4. 중복 헤더명 때문에 JSON 단계에서 컬럼이 덮어써짐

`to_sections()`는 각 행을 dict로 만들 때 헤더명을 key로 그대로 사용한다.

- `row_dict[key] = cell`
- 근거: `ps-docparser/extractors/bom_converter.py:61`, `:95`

이번 표의 헤더에는 같은 이름이 여러 번 반복된다.

- `자재중량 [Kg]` 4회
- `자재면적 [m2]` 4회

dict는 동일 key를 하나만 유지하므로, 뒤쪽 값이 앞쪽 값을 덮어쓴다. 이 때문에 `UNIT`, `LOSS`, `WEIGHT` 등 복수 보조열 정보가 정상 보존되지 않는다.

여기서 중요한 보강 해석이 하나 더 있다.

- 이 손실은 **JSON에서 끝나지 않는다**
- 현재 파이프라인은 JSON 구조를 기반으로 Excel을 만들기 때문에, JSON에서 사라진 보조열 정보는 Excel도 복구하지 못한다
- 즉 F4는 **JSON 단독 문제가 아니라 JSON과 Excel을 동시에 오염시키는 상위 손실 지점**이다

### F5. Excel이 `Table` 시트 1개만 만든 것은 현재 분류 로직상 예상 가능한 결과

현재 Excel 분류기는 헤더 패턴으로만 시트 타입을 구분한다.

- `estimate`: `명칭 + 금액`
- `detail`: `품명 + 합계_금액`
- `condition`: `일반사항/특기사항`
- 나머지: `generic`
- 근거: `ps-docparser/exporters/excel_classifier.py:19`

또한 generic 분기 자체는 아래 경로를 탄다.

- generic 적재: `ps-docparser/exporters/excel_exporter.py:135`
- 메타 시트 분기: `ps-docparser/exporters/excel_exporter.py:194-198`
- generic 시트명 `Table` 생성: `ps-docparser/exporters/excel_exporter.py:205-212`

이번 JSON의 최종 테이블 헤더는 `DESCRIPTION`, `DWG NO.`, `MAT'L`, `SIZE`, `수량`, `단위`, ... 이므로, `estimate/detail/condition` 어느 쪽에도 맞지 않아 `generic`으로 분류된다.

그래서 Excel에는 `"견적서"`, `"내역서"`, `"조건"` 대신 `"Table"` 시트만 생성되었다.

이 역시 **현재 코드 기준으로는 예상 가능한 결과**다. 다만 사용자가 시트명만 보고 내용을 이해하기 어렵다는 점에서, 출력 표현 개선 후보이기도 하다.

## 5. 원인 종합

이번 이상 현상은 하나의 버그가 아니라 아래 3개가 겹친 결과다.

### 원인 A. 테스트 문서와 BOM 프리셋의 부적합

현재 테스트 문서는 순수 BOM 도면이 아니라 견적 문서에 가깝다.  
따라서 BOM 프리셋은 문서 전체를 풍부하게 내보내지 않고, BOM 유사 표만 좁게 잡는다.

### 원인 B. 메타 추출 대상 자체가 현재 원문에 없음

Phase 14 메타 추출기는 title block 라인을 대상으로 설계되어 있다.  
하지만 이번 MD 원문에는 그 패턴이 거의 없고, `DWG NO.`도 메타 라인이 아니라 표 헤더다.

### 원인 C. 복합 헤더 HTML 테이블 대응 미비

마지막 자재표는 2행 헤더 + 중복 컬럼명을 가진다.  
현재 BOM HTML 파서는 이를 단일 헤더 행으로만 해석하고, JSON dict 변환은 중복 key를 허용하지 않는다.

### 원인 간 관계 보강 메모

A/B/C는 서로 다른 층위의 원인이다.

- A는 **문서-프리셋 미스매치**다
- B는 **입력 원문에 필요한 메타 패턴이 없는 데이터 조건**이다
- C는 **파서/컨버터 구현 한계**다

특히 C와 F4는 분리해서 볼 수 있지만, 실제 영향은 직렬이다.

- C가 복합 헤더를 잘못 해석한다
- 그 상태에서 F4가 중복 헤더 overwrite를 일으킨다
- 그 결과가 JSON에 굳어지고 Excel이 그대로 상속한다

즉 **이번 손실 체인의 실제 본체는 C + F4의 조합**이다.

### 샘플 일반화 가능성 메모

현재 이 결론은 **문제 문서 1건 실측**을 기준으로 한다. 따라서 아래를 구분해야 한다.

- 이 문서가 예외적인 혼합 문서 1건인지
- 아니면 실제 업무 문서에서 자주 나오는 `견적 + BOM` 혼합 패턴인지

의사결정 기준은 아래처럼 두는 것이 적절하다.

- 동일한 패턴 문서가 1건 더 나오면: parser 보강 우선
- 동일한 패턴 문서가 3건 이상 반복되면: preset 경고나 라우팅 개선까지 구조 변경 검토
- 현재 1건만 확인된 상태라면: parser 보강은 정당하지만, 자동 라우팅은 아직 과감한 결정일 수 있음

## 6. 코드 동작상 예상 가능한 부분 vs 제품 관점에서 수정이 필요한 부분

### 코드 동작상 예상 가능한 부분

- 파일 생성 자체는 정상
- BOM 프리셋이 BOM 유사 테이블만 뽑은 것은 현재 설계와 일치
- `"도면_메타"` 시트가 안 생긴 것은 추출 결과가 전부 `None`이므로 현재 코드와 일치
- `"Table"` 시트 1개만 생성된 것도 `generic` 분류 결과와 일치

### 제품/사용자 기대 기준으로 수정이 필요한 부분

- 복합 헤더(2행 헤더) 테이블 파싱
- 중복 헤더명 보존 방식
- generic 시트명 의미화 또는 BOM 자재표 인지성 개선
- 문서 종류에 맞는 프리셋 선택 또는 최소한의 경고 로그
- 필요 시 Phase 14 메타 추출 범위를 표 내부 `DWG NO.` 컬럼까지 확장할지 정책 결정

### 해석 주의

여기서 `코드 동작상 예상 가능`을 `수정 불필요`로 읽으면 안 된다.

- F1은 코드 기준으로는 예상 가능
- 그러나 사용자가 문서 전체 구조를 기대했다면 제품 기준으로는 개선 필요

즉 이 문서는 `버그/비버그`만 가르는 문서가 아니라,
**“어떤 것은 코드 설명으로 끝나고, 어떤 것은 제품 개선으로 이어져야 하는가”를 구분하는 문서**로 읽어야 한다.

### 7.2 추천 실행 순서

#### 1단계. 즉시 착수 가능한 핫픽스

- P1 `중복 헤더 dedupe`

이유:

- 변경량이 가장 작다
- JSON 단계 손실을 바로 멈출 수 있다
- Excel 손실도 동시에 줄어든다

#### 2단계. 구조적 정상화

- P2 `다단 헤더 병합`

이유:

- 지금 가장 눈에 띄는 “가짜 데이터 1행” 문제를 해결한다
- P1 없이도 의미가 있지만, P1과 함께 해야 완결된다

#### 3단계. 사용자 인지성 개선

- P3 `generic BOM 시트명 의미화`
- P4 `BOM 프리셋 경고 로그`

이유:

- 데이터 자체를 바꾸지 않고도 사용자가 결과를 이해하기 쉬워진다
- 혼합 문서를 BOM 프리셋으로 돌렸을 때 왜 이런 결과가 나왔는지 설명력이 생긴다

#### 4단계. 정책성 기능

- P5 `table 기반 dwg_no fallback`
- P6 `자동 라우팅 / 듀얼 파이프라인 머지`

이유:

- 둘 다 기능적 의미는 크지만, 제품 정책과 오탐 리스크가 동반된다
- 따라서 핫픽스와 분리하는 것이 안전하다

### 7.3 바로 착수 가능한 “작업 계획서” 형태로 읽는 법

이번 보고서의 권장 조치는 단순 우선순위가 아니라, 아래 3가지를 함께 포함하도록 보강했다.

- 영향 파일/핵심 위치
- 예상 수정 규모
- 회귀 검증 시나리오

따라서 이 문서는 순수 분석서이면서, 동시에 **Phase 15 작업 계획서의 초안**으로도 사용할 수 있다.

## 8. 최종 결론

이번 테스트 결과가 기대와 다르게 나온 주된 이유는 **Phase 14 기능이 완전히 죽어서가 아니라**,  
**현재 테스트 문서가 BOM 도면 전제와 다르고, 마지막 BOM 유사 테이블도 복합 헤더 구조라 현재 파서가 충분히 보존하지 못했기 때문**이다.

한 줄로 요약하면:

> **문서 선택은 BOM 프리셋과 어긋나 있고, 남은 1개 자재표는 복합 헤더 + 중복 헤더 overwrite 때문에 JSON과 Excel이 함께 깨졌다.**

따라서 다음 작업은 Phase 14 메타 추출기 자체를 바로 의심하기보다 아래 순서로 가는 것이 가장 합리적이다.

1. `bom_converter.py`의 중복 헤더 overwrite 차단
2. `bom_table_parser.py`의 복합 헤더 병합
3. `excel_exporter.py`의 generic BOM 시트명 개선
4. 혼합 문서 경고 로그 추가
5. 필요 시 메타 fallback / 자동 라우팅을 별도 Phase로 분리

최종 권고:

> **지금 가장 먼저 고쳐야 할 것은 “메타 추출기”가 아니라, “복합 헤더 파서 + 중복 헤더 overwrite”다.**
