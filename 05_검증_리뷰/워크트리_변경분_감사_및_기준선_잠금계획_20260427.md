### 4.6 중복 샘플 PDF

관련 파일:

- `01_OCR_결과물/53-83 OKOK.pdf`
- `00_견적서_원본/53-83 OKOK.pdf`

감사 판단:

- 현재 워크트리의 유일한 삭제 표시는 `01_OCR_결과물/53-83 OKOK.pdf`다.
- 사용자는 이 파일을 `00_견적서_원본/53-83 OKOK.pdf`와 동일한 샘플로 취급하라고 지시했다.
- 해시 확인 결과, 삭제된 Git blob과 `00_견적서_원본/53-83 OKOK.pdf`의 blob hash가 동일하다.
- 따라서 이 삭제 표시는 데이터 유실 위험이 아니라 중복 샘플 위치 정리로 취급한다.
- 단, `00_견적서_원본/`은 `.gitignore`로 제외된 로컬 원본 폴더다. 따라서 Git 저장소 안에 이 PDF 샘플을 계속 추적해야 하는 정책이라면 삭제 커밋 전에 별도 판단이 필요하다.

처리 원칙:

- 로컬 실측 기준 샘플은 `00_견적서_원본/53-83 OKOK.pdf`로 유지한다.
- `01_OCR_결과물/53-83 OKOK.pdf` 삭제는 코드 기준선 위험에서 제외한다.
- 커밋 시에는 기능 변경과 섞지 말고 중복 샘플 정리로 명시한다.
- Git 저장소 자체에 PDF 샘플을 남길 필요가 있으면 `01_OCR_결과물/` 삭제를 되돌리거나, 별도 fixture 보관 정책을 세운다.

## 7. 커밋 또는 잠금 단위 제안

커밋은 사용자가 명시적으로 요청하기 전에는 실행하지 않는다. 다만 잠금 단위는 아래처럼 나누는 것이 안전하다.

1. `docs: record backend stabilization baseline`
   - 기준선 문서, README, 감사 보고서
2. `fix: stabilize BOM table headers and sparse rows`
   - BOM parser/converter/fixture/regression
3. `feat: add output contracts and manifest-backed QA`
   - validator, run manifest, analyzer, Excel role 우선 분류, aggregation
4. `feat: formalize BOM fallback and auto-route behavior`
   - CLI option, pipeline, detector, smoke tests
5. `feat: add Gemini key rotation support`
   - config, factory, gemini engine, rotator tests
6. 별도 정리: `01_OCR_결과물/53-83 OKOK.pdf` 삭제
   - `00_견적서_원본/53-83 OKOK.pdf`와 동일 샘플이므로 데이터 유실 위험은 낮음
   - 단, `00_견적서_원본/`은 Git ignore 대상이므로 Git에 PDF 샘플을 계속 둘 필요가 있는지는 별도 판단
   - 커밋 시 중복 샘플 정리로 명시

## 8. 남은 위험

- 전체 `pytest`는 이번 실행에서 통과했다. 다만 커밋 직전에는 워크트리 변경 여부를 다시 확인하는 것이 안전하다.
- 실측 suite는 `실측테스트_20260427_baseline_lock`으로 새로 재실행했다. 이 산출물은 `output/` 아래에 있으므로 Git 추적 대상인지 여부를 커밋 전 다시 확인한다.
- `detector.py`, `document_pipeline.py`, `bom_table_parser.py` 변경량이 크므로 전체 회귀 전에는 완전 잠금으로 보지 않는다.
- Google Drive 동기화 경로이므로 파일 삭제/충돌 파일/임시 산출물은 커밋 전 다시 확인해야 한다.
- `.env`는 열람하지 않았다. 커밋 전 secret 포함 여부는 파일 목록 기준으로 별도 확인해야 한다.
- `01_OCR_결과물/53-83 OKOK.pdf` 삭제 표시는 `00_견적서_원본/53-83 OKOK.pdf`와 동일 샘플임을 확인했다. 데이터 유실 위험은 낮지만, `00_견적서_원본/`은 Git ignore 대상이므로 Git 추적 샘플 보존 여부는 별도 판단해야 한다.

## 9. 결론

현재 변경분은 무작위 수정이 아니라 2026-04-24 백엔드 안정화 기준선과 대체로 일치한다.

다만 잠금 순서는 중요하다. `53-83 OKOK.pdf` 삭제 표시는 `00_견적서_원본/`의 동일 샘플을 로컬 기준본으로 두는 중복 샘플 정리로 분리하고, BOM parser/output contract/manifest/analyzer 묶음을 먼저 고정한 뒤 routing/fallback과 Gemini key rotation을 별도 단위로 검토하는 것이 안전하다.

현재 바로 잠금 가능한 최소 기준은 다음이다.

- `git diff --check`: PASS
- 기준선 관련 집중 테스트: `93 passed`
- 전체 회귀 테스트: `753 passed, 6 skipped`, coverage `81.52%`
- 기존 실측 QA 산출물: header/key mismatch 0, bad composite headers 0, manifest representative/diagnostic 분리 확인
- 신규 실측 QA 산출물: `실측테스트_20260427_baseline_lock`, header/key mismatch 0, bad composite headers 0, manifest representative/diagnostic 분리 확인
- `53-83 OKOK.pdf` 샘플 동일성: 삭제된 `01_OCR_결과물/` blob과 `00_견적서_원본/` 현재 파일 hash 일치. 단, `00_견적서_원본/`은 Git ignore 대상

최종 기준선 잠금 전 남은 핵심 판단은 삭제 표시 PDF의 복구 여부가 아니라, Git 추적 샘플을 계속 둘지와 커밋 단위를 기능 기준선/중복 샘플 정리로 어떻게 나눌지다.

## 10. 프론트엔드 전환 상세 구현 기술서

### 10.4 공통 API 목록

Phase15 v1 API는 아래 7개로 고정한다.

```text
GET  /api/config
POST /api/jobs
GET  /api/jobs/{job_id}
POST /api/jobs/{job_id}/cancel
GET  /api/jobs/{job_id}/artifacts
GET  /api/jobs/{job_id}/artifacts/{artifact_id}
GET  /api/jobs/{job_id}/qa
```

엔진 옵션은 `auto`, `zai`, `gemini`, `local`, `mistral`, `tesseract`로 둔다. `auto`는 CLI에 `--engine`을 전달하지 않고 기존 config 기본 엔진을 사용한다.
