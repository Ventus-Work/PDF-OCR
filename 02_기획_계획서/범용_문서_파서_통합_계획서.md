# 범용 문서 파서 통합 계획서 (`ps-docparser`)

## 배경 및 목적

현재 3개의 독립 코드베이스가 **서로 다른 폴더에 흩어져** 있고, 각각 고유한 문서 유형만 처리할 수 있다. 이를 **하나의 범용 CLI/GUI 파이프라인**으로 통합하여, PDF/HWPX 등 어떤 문서든 던지면 정형 데이터(JSON/Excel)가 나오는 올인원 도구를 만든다.

---

## 현황 분석 (AS-IS)

### A. `ocr.py` (2,310줄 / tkinter GUI)

| 항목 | 내용 |
|---|---|
| **역할** | PDF/이미지 → OCR → Excel/Word 저장 (GUI 앱) |
| **OCR 엔진** | Z.ai GLM-OCR (유료 API) + Mistral OCR (보조) + Tesseract (3차 폴백) |
| **특화 기능** | 배관 BOM/LINE LIST 추출, 한국어 거래명세표 파싱 |
| **입력** | PDF, PNG, JPG, 스크린샷(클립보드) |
| **출력** | Excel(.xlsx), Word(.docx) |
| **장점** | GUI가 있어 비개발자도 사용 가능. BOM 3단계 폴백(Z.ai→Mistral→Tesseract) |
| **단점** | 코드가 단일 파일 2,300줄에 몰아넣기(God Object). Z.ai 유료 API 종속. 품셈 문서 미지원 |

### B. `pdf_extractor/` (3개 파일 / CLI+GUI)

| 항목 | 내용 |
|---|---|
| **역할** | PDF → 구조화 마크다운(.md) 변환 (1단계 추출 엔진) |
| **OCR 엔진** | pdfplumber(텍스트/무료) + Gemini Vision(테이블/유료) 하이브리드 |
| **특화 기능** | 목차(TOC) 연동, PAGE/SECTION/CONTEXT 마커 자동 삽입, bbox 보정 |
| **입력** | PDF + 목차파일(.json/.txt) |
| **출력** | 마크다운(.md) — HTML `<table>` 태그 포함 |
| **장점** | 표 파싱 품질 최고. 비용 효율적(텍스트=무료, 표만 API). 페이지 범위 지정 |
| **단점** | 출력이 마크다운 중간산물이라 바로 Excel/DB로 못 넣음. 품셈 전용 도메인에 최적화 |

### C. `standalone_parser/` (3개 파일 / CLI)

| 항목 | 내용 |
|---|---|
| **역할** | 마크다운(.md) → 구조화 JSON 변환 (2단계 정제 엔진) |
| **OCR 엔진** | 없음 (**파서만**, 1단계 결과물을 입력으로 받음) |
| **특화 기능** | rowspan/colspan 전개, 주석 분리, 할증 조건 추출, 교차참조 감지, 테이블 타입 분류 |
| **입력** | 마크다운(.md) + 목차 JSON |
| **출력** | Python dict / JSON |
| **장점** | 표 구조 완벽 복원. 도메인 메타데이터(주석/조건/참조) 자동 추출 |
| **단점** | 단독 실행 불가(1단계 결과물 필요). 품셈 도메인 전용 |

## 현재 흐름의 문제점

```
[현재: 파편화된 3개 도구]

ocr.py (BOM/거래명세표)      ──→  Excel
pdf_extractor (품셈 PDF)     ──→  마크다운.md  ──→  standalone_parser  ──→  JSON
```

> ⚠️ **핵심 문제:** 사용자가 문서 유형에 따라 **어떤 도구를 쓸지 직접 판단**해야 하고, pdf_extractor→standalone_parser는 수동으로 파일을 전달해야 한다. 범용성 제로.

### PDF 유형별 처리 전략

| PDF 유형 | 감지 방법 | 동작 | 경고 |
|---|---|---|---|
| 🟢 텍스트+표 | pdfplumber로 텍스트 추출 가능 | **기본 경로**: pdfplumber(텍스트) + AI엔진(표) | 없음 |
| 🟡 이미지 포함 | 페이지 내 이미지 객체 감지 | **OCR 라우터**: 이미지 영역만 AI 엔진으로 처리 | `⚠️ 이미지 포함 — OCR 엔진 사용` |
| 🔴 스캔본 | 텍스트 추출량 ≈ 0 | **전면 OCR**: 전체 페이지를 이미지로 변환 후 OCR | `⚠️ 스캔 문서 — 전면 OCR, 비용 발생` |

## 설계 원칙

### 원칙 1: 이식성 (Portability)
폴더째로 복사/Git push/USB 이동해도 **그대로 동작**해야 한다.

| 규칙 | 설명 |
|---|---|
| **절대경로 금지** | 모든 경로는 `__file__` 기준 상대경로로 처리 |
| **자기완결형** | `.env`, `cache/`, `requirements.txt` 전부 프로젝트 폴더 안에서 해결 |
| **외부 의존 경로 없음** | 시스템 전역 설정에 의존하지 않음 (Poppler 등은 `.env`로 설정) |

```python
# 올바른 경로 처리 예시
BASE_DIR = Path(__file__).resolve().parent
CACHE_DIR = BASE_DIR / "cache"
ENV_PATH = BASE_DIR / ".env"
```

### 원칙 2: 범용성 (Universal)
건설 품셈, BOM이 아닌 **모든 종류의 PDF를 보기 좋은 파일로 변환**하는 것이 핵심 목적이다.

| 항목 | 기존 (잘못됨) | 수정 (확정) |
|---|---|---|
| 기본 동작 | 품셈 특화 | **범용 변환 (아무 PDF → 깔끔한 MD/Excel)** |
| 품셈/BOM | 기본 모드 | **프리셋 옵션** (`--preset pumsem/bom`) |
| `detector.py` | 필수 자동 판별 | 보조 기능, 기본은 범용 |

### 원칙 3: 엔진 선택제 (Strategy Pattern)
OCR/AI 엔진을 플러그인처럼 교체할 수 있어야 한다.

```python
# 공통 인터페이스
class BaseExtractor:
    def extract_text(self, page) -> str: ...
    def extract_table(self, image) -> str: ...

# 엔진별 구현
class GeminiExtractor(BaseExtractor): ...   # Gemini Vision
class ZaiExtractor(BaseExtractor): ...      # Z.ai GLM-OCR
class LocalExtractor(BaseExtractor): ...    # pdfplumber only (무료)
```

```bash
# CLI에서 엔진 선택
python main.py "파일.pdf" --engine gemini     # Gemini Vision
python main.py "파일.pdf" --engine zai        # Z.ai GLM-OCR
python main.py "파일.pdf" --engine local      # 로컬 전용 (무료)
```

`.env` 파일에서 기본 엔진 설정:
```
DEFAULT_ENGINE=gemini
GEMINI_API_KEY=AIzaSy...
ZAI_API_KEY=abc123...
```

### 원칙 4: 모듈 분리 + 상세 주석 (Single Responsibility)
한 파일에 여러 기능을 몰아넣지 않는다. **기능별로 개별 파일에 함수화**하고, `main.py`는 조립만 한다.

| 규칙 | 설명 |
|---|---|
| **1기능 = 1모듈** | 각 기능은 독립 파일로 분리. 단독 테스트 가능해야 함 |
| **main.py는 컨트롤러** | import해서 파이프라인을 조립하는 역할만. 비즈니스 로직 금지 |
| **Why 주석 필수** | 각 함수에 "왜 이 로직인지", 입력/출력/의존성을 주석으로 기록 |
| **반례: ocr.py** | 2,300줄 단일 파일 = 수정 불가, 테스트 불가, 재사용 불가 |

```python
# ❌ 잘못된 예 (ocr.py 현재 구조)
class MistralOCRApp:
    def perform_ocr(self): ...        # OCR
    def extract_bom_tables(self): ... # 파싱
    def save_to_excel(self): ...      # 저장
    def run(self): ...                # GUI
    # → 2,300줄 God Object

# ✅ 올바른 예 (ps-docparser 구조)
# main.py — 조립만
from extractors.hybrid_extractor import extract_pdf
from parsers.table_parser import parse_tables
from exporters.excel_exporter import export_to_excel

result = extract_pdf("input.pdf", engine="gemini")
tables = parse_tables(result)
export_to_excel(tables, "output.xlsx")
```

```python
# 각 함수 파일의 주석 규칙 예시 (extractors/hybrid_extractor.py)
def extract_pdf(pdf_path: str, engine: str = "gemini") -> str:
    """
    PDF에서 텍스트+표를 추출하여 구조화 마크다운을 반환한다.
    
    Why: pdfplumber만으로는 표 구조(셀 병합/테두리 없는 표)를 복원 못함.
         AI 엔진(Gemini/Z.ai)으로 표 이미지를 HTML로 변환하여 보완.
    
    Args:
        pdf_path: PDF 파일 경로 (상대/절대 모두 가능)
        engine: 사용할 AI 엔진 ("gemini" | "zai" | "local")
    
    Returns:
        str: 구조화 마크다운 (HTML <table> 포함)
    
    Dependencies:
        - pdfplumber (텍스트 추출)
        - pdf2image + Poppler (페이지 이미지화)
        - google-generativeai 또는 zhipuai (AI 엔진)
    """
```
