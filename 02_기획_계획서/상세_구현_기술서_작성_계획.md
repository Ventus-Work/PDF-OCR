### 핵심 설계 포인트

1. **BaseExporter ABC**: `export(sections, output_path)` 공통 시그니처, `_prepare()` → `_write()` → `_finalize()` 템플릿 메서드 패턴
2. **json_exporter**: 현재 main.py L282~310에 직접 작성된 JSON 저장 로직을 분리
3. **estimate preset**: pumsem.py와 동일한 구조 (TABLE_TYPE_KEYWORDS, PARSE_PATTERNS, SHEET_CONFIG)
4. **detector.py**: MD 텍스트 키워드 빈도 기반, 프리셋 미지정 시에만 동작, 확신도 낮으면 "general" 반환
5. **한글 균등배분 병합 (K1)**: `"품   명"` → `"품명"` 변환. 한글 1글자 토큰 70%+ 비율 감지 + fontSize 기반 간격 판정. kordoc `cluster-detector.ts` 알고리즘 Python 포팅
6. **Bold-fake 제거 (K4)**: 동일 텍스트 + Y좌표 동일 + X좌표 차이 ≤ 3px → 중복 판정, 첫 번째만 유지

#### 문제 D: HTML 테이블 파싱 3중 중복 (치명적)

동일한 HTML `<table>` 파싱이 3곳에서 **각각 다른 방식**으로 구현됨:

| 위치 | 함수 | 파싱 방법 |
|------|------|----------|
| L692-709 | `extract_korean_business_tables` | 수동 정규식 (`re.findall`) |
| L1131-1144 | `parse_table_from_text` | 수동 정규식 (거의 동일 복붙) |
| L1331-1348 | `extract_bom_tables` | `pd.read_html()` (완전히 다른 방식) |

**문제점:** 하나를 수정하면 나머지 2곳도 수정해야 하는데 빠뜨릴 가능성이 높고, 3곳의 파싱 결과가 미묘하게 다를 수 있음.

#### 문제 E: 테이블 분기 3중 복붙 (주의)

동일한 if-elif-else 분기가 3곳에서 완전 복붙:

```python
# L973-978 (save_to_excel), L1189-1194 (save_to_word), L1914-1919 (process_batch)
if self.bom_ocr.get():
    tables = self.extract_bom_tables(text)
elif self.enhanced_ocr.get():
    tables = self.extract_korean_business_tables(text)
else:
    tables = self.parse_table_from_text(text)
```

Strategy Pattern으로 해결해야 할 전형적인 케이스.

### 5.8 외부 참조: kordoc 라이브러리 알고리즘 분석

#### 개요

[kordoc](https://github.com/chrisryugj/kordoc) (v2.2.6, 721 stars, MIT 라이선스)은 TypeScript/Node.js 기반 한국 공공문서 파서다. HWP/HWPX/PDF/XLSX/DOCX → Markdown 변환을 지원한다. Python 프로젝트에 직접 import는 불가하나, MIT 라이선스이므로 **알고리즘 차용(Python 포팅)이 법적으로 가능**하다 (저작권 고지 필요).

#### 우리 파이프라인(pdfplumber)과의 기술 비교

| 항목 | pdfplumber (현재 우리) | kordoc |
|------|----------------------|--------|
| **기반 라이브러리** | pdfminer.six | pdfjs-dist (Mozilla PDF.js) |
| **선 추출** | pdfminer의 LTCurve 객체 | pdfjs operatorList 직접 파싱 |
| **좌표 허용 오차** | 고정 상수 (`snap_tolerance=3`, `join_tolerance=3`) | **동적 — 교차점 선폭(lineWidth)에 비례하여 자동 계산** |
| **선 없는 테이블** | `text` 전략 별도 설정 필요 (기본 미감지) | **자동 폴백** (`cluster-detector`: 텍스트 정렬 패턴 기반) |
| **한글 균등배분** | 처리 없음 | **전용 처리** (`"품   명"` → `"품명"`, fontSize 기반 간격 판정) |
| **Bold-fake 제거** | 처리 없음 | **±3px 중복 글리프 자동 제거** |
| **레이아웃 테이블** | 사용자가 판단 | **자동 감지/해체** (`flattenLayoutTables`) |
| **다단 레이아웃** | 없음 | **재귀적 XY-Cut** 알고리즘 |
| **헤더/푸터 제거** | 없음 | **패턴 반복 기반 자동 제거** |
| **병합 셀 감지** | 내부 처리 (결과만 반환) | **내부 경계선 커버리지 75% 임계값** |
| **텍스트→셀 매핑** | 내부 처리 | **교차 면적 비율 0.3 임계값 + 행 그룹핑** |
| **병합 셀 출력** | 평문 2D 배열 | **HTML `<table>` with colspan/rowspan** |

#### 차용 가치 판정 — 5개 알고리즘

##### 알고리즘 K1: 한글 균등배분 병합 (우선도: 최상)

한국 공문서/견적서에서 빈번한 문제. 현재 파이프라인에 **전혀 없는 기능**.

```
PDF 원문:  "현 장 대 응 단 장"   "품   명"   "규   격"
필요 결과: "현장대응단장"          "품명"       "규격"
```

kordoc 구현 방식 (`cluster-detector.ts`):
1. 한글 1글자 토큰(`/^[가-힣\d]$/`) 비율 70%+ 이면 균등배분으로 판정
2. fontSize 기반 간격 임계값 계산 (font폭 × 1.5 이내 → 같은 단어)
3. 연속 한글 1글자 토큰을 하나의 문자열로 합침

**적용 위치:** `parsers/text_cleaner.py` 또는 `utils/text_formatter.py`에 `merge_spaced_korean()` 함수 추가
**적용 시점:** Phase 3-B (즉시)
**예상 규모:** ~30줄

##### 알고리즘 K2: 선 없는 테이블 자동 감지 (우선도: 상)

현재 `table_utils.py:detect_tables()`에서 pdfplumber `find_tables()`가 실패하면, `hybrid_extractor.py:176-181`에서 **전체 페이지를 Gemini Vision에 전송**한다 (비용 발생). kordoc의 `cluster-detector.ts` 알고리즘을 폴백으로 넣으면 **AI 호출 없이** 테이블을 잡을 수 있는 케이스가 늘어남.

kordoc 구현 방식 (`cluster-detector.ts`):
1. Y좌표 행 그룹핑 (`Y_TOL = 3pt` 허용)
2. 헤더 행 감지: 2~6개 짧은 한글 아이템 + 큰 갭 + 넓은 X 범위
3. X좌표 열 클러스터링 (`COL_CLUSTER_TOL = 15pt`)
4. 행별 적응형 갭 분석 (희소 행: 12pt 임계, 밀집 행: median×2.5 임계)
5. 다중행 셀 병합: Y갭이 작고 아이템 수가 적은 연속 행을 이전 행에 합병

**적용 위치:** `extractors/table_utils.py`에 `detect_tables_by_text_alignment()` 함수 추가, `detect_tables()` 실패 시 폴백
**적용 시점:** Phase 4
**예상 규모:** ~200줄

##### 알고리즘 K3: 동적 허용 오차 시스템 (우선도: 중)

pdfplumber의 `find_tables()`에 전달하는 `table_settings`의 `snap_tolerance`, `join_tolerance`가 고정 상수(3)인 것을 동적 계산으로 개선.

kordoc 구현 방식 (`line-detector.ts`):
```
vertex_radius = max(lineWidth_h, lineWidth_v) × VERTEX_MERGE_FACTOR(4)
coord_merge_tol = max(MIN_COORD_MERGE_TOL(8), vertex_radius)
```

**적용 위치:** `extractors/table_utils.py`의 `detect_tables()` 내부에서 pdfplumber 호출 전 선 두께 사전 분석 → 동적 `snap_tolerance` 설정
**적용 시점:** Phase 4
**예상 규모:** ~40줄

##### 알고리즘 K4: Bold-fake 글리프 중복 제거 (우선도: 중)

일부 PDF에서 볼드를 ±3px 오프셋 중복 텍스트로 구현하여, 추출 시 `"품명품명"` 같은 중복이 발생.

kordoc 구현 방식 (`parser.ts`):
1. 동일 텍스트 + Y좌표 동일 + X좌표 차이 ≤ 3px → 중복 판정
2. 중복 글리프 중 첫 번째만 유지, 나머지 제거

**적용 위치:** `extractors/text_extractor.py` 또는 `utils/text_formatter.py`
**적용 시점:** Phase 3-B (즉시)
**예상 규모:** ~20줄

##### 알고리즘 K5: 레이아웃 테이블 자동 해체 (우선도: 하)

테이블 형태이지만 실제로는 페이지 레이아웃용인 구조를 자동 감지하여 표가 아닌 일반 텍스트로 변환.

```
┌────────────────────┬──────────────┐
│ 제출처: 고려아연    │ 견적번호: 123│  ← 데이터 테이블이 아닌 레이아웃
└────────────────────┴──────────────┘
```

kordoc 구현 방식 (`parser.ts`):
- 테이블이 페이지 전체 너비의 90%+ 차지 + 내부 셀이 자유 텍스트(숫자 열 없음) → 레이아웃으로 판정

**적용 위치:** `exporters/excel_exporter.py`의 `_classify_table()` 또는 `parsers/table_parser.py`
**적용 시점:** Phase 4 이후
**예상 규모:** ~50줄

## 6. Phase 5 기술서 범위 (캐싱 + GUI + 배치처리)

### 목적
API 비용 절감을 위한 캐싱, 비개발자를 위한 GUI, 폴더 일괄 처리 기능을 추가한다.

### 구현 항목

| # | 파일 | 작업 | 원본 소스 | 비고 |
|---|------|------|----------|------|
| 1 | `cache/table_cache.py` | 테이블 이미지 해시 → 결과 캐싱 (SQLite) | 신규 설계 | API 비용 절감 핵심 |
| 2 | `cache/table_cache.db` | SQLite DB 파일 (자동 생성) | - | |
| 3 | `gui.py` | tkinter GUI (파일 선택 + 옵션 + 진행바) | ocr.py `setup_gui()` L381~524 | 선택사항 |
| 4 | `main.py` 확장 | `--batch` 폴더 일괄 처리 | ocr.py `process_batch()` L1881~2303 | |
| 5 | 캐시 통합 | extractors에 캐시 레이어 주입 | 기존 확장 | |

### 핵심 설계 포인트

1. **캐시 키**: `sha256(이미지 바이트 + 엔진명 + 프롬프트)` → 동일 표 이미지는 API 재호출 안 함
2. **캐시 TTL**: 기본 30일, `.env`에서 설정 가능
3. **GUI**: ocr.py의 tkinter 코드를 리팩터하되, 비즈니스 로직은 `main.py`의 함수를 호출하는 얇은 UI 레이어
4. **배치**: `--batch "폴더/"` → glob으로 PDF 수집 → 순차 처리 + 요약 리포트

## 7. 기술서 작성 우선순위 및 실행 순서

```
┌───────────────────────────────────────────────────────┐
│  1순위: Phase 3-B 기술서 작성 → 구현 → 검증           │
│         (기존 Phase3 기술서 80% 재활용 가능)            │
│         예상: 기술서 1일 + 구현 1~2일                   │
├───────────────────────────────────────────────────────┤
│  2순위: Phase 4 기술서 작성 → 구현 → 검증             │
│         (ocr.py 신규 설계, 가장 큰 작업)               │
│         예상: 기술서 2일 + 구현 3~4일                   │
├───────────────────────────────────────────────────────┤
│  3순위: Phase 5 기술서 작성 → 구현 → 검증             │
│         (선택사항 포함, 필수는 캐싱만)                  │
│         예상: 기술서 1일 + 구현 1~2일                   │
└───────────────────────────────────────────────────────┘
```

### 의존 관계

```
Phase 3-B ──→ Phase 4 ──→ Phase 5
   │              │            │
   │              │            └─ gui.py는 모든 기능 완성 후
   │              └─ bom_extractor는 base_engine OCR 인터페이스 필요 (3-B)
   └─ base_exporter ABC + detector.py가 Phase 4의 기반
```

- **Phase 3-B 선행 필수 이유**: `base_exporter.py` ABC가 Phase 4에서 BOM용 exporter 확장 시 기반이 됨. `detector.py`가 Phase 4의 BOM 감지에 활용됨.
- **Phase 4 → Phase 5 순서**: 캐싱은 API 호출이 있는 Phase 4 엔진에 적용되므로, 엔진 구현 후 캐시 레이어를 씌우는 것이 자연스러움.
