# Phase15-Step3 결과 QA 조회 다운로드 상세 구현 기술서

작성일: 2026-04-27

대상 프로젝트: `Project/PJT_A/ps-docparser`

## 1. 목적

job 실행 결과로 생성된 산출물을 API와 UI에서 안전하게 탐색, 요약, 다운로드할 수 있게 한다.

Step3의 핵심은 `RUN_MANIFEST.json`, `RUN_SUMMARY.md`, `OUTPUT_QA_REPORT.md`, md/json/xlsx 파일을 안정적으로 읽는 artifact service다.

## 2. 구현 대상

주요 파일:

```text
api/artifacts.py
api/schemas.py
api/app.py
tests/unit/api/test_artifacts.py
```

대상 output:

```text
output/ui_runs/<job_id>/result/
```

읽을 수 있는 파일:

```text
*.md
*.json
*.xlsx
RUN_MANIFEST.json
RUN_SUMMARY.md
OUTPUT_QA_REPORT.md
```

## 3. 제외 범위

- 파일 내용 편집
- Excel preview 렌더링
- 대형 파일 streaming 최적화
- 결과 비교 diff UI
- output 폴더 전체 탐색

## 4. ArtifactItem 모델

필드:

```text
artifact_id: string
name: string
relative_path: string
kind: md | json | xlsx | manifest | summary | qa | other
size_bytes: int
download_url: string
role: representative | diagnostic | unknown
domain: bom | estimate | pumsem | trade_statement | generic | unknown
quality_status: ok | warning | fail | unknown
```

`artifact_id` 규칙:

- result_dir 기준 relative path를 URL-safe base64 또는 안정 hash로 변환
- 사용자가 임의 path를 직접 넣지 않게 한다.

## 5. Artifact 탐색 규칙

1. job의 result_dir를 확인한다.
2. result_dir 아래 파일만 재귀 탐색한다.
3. `RUN_MANIFEST.json`, `RUN_SUMMARY.md`, `OUTPUT_QA_REPORT.md`를 우선 노출한다.
4. 일반 산출물은 확장자별로 분류한다.
5. manifest에 나온 primary/diagnostics 정보를 artifact metadata에 병합한다.
6. manifest에 없는 파일은 `role=unknown`, `domain=unknown`으로 표시한다.

## 6. QA 요약 모델

`GET /api/jobs/{job_id}/qa` 응답:

```text
job_id: string
status: ok | warn | fail | unknown
json_files: int
excel_files: int
has_manifest: bool
manifest_inputs: int
manifest_representative: int
manifest_diagnostic: int
header_key_mismatch: int
bad_composite_headers: int
quality_warnings: dict[str, int]
manifest_domains: dict[str, int]
report_path: string | null
summary_markdown: string | null
```

기준:

- `OUTPUT_QA_REPORT.md`가 있으면 우선 파싱한다.
- `RUN_MANIFEST.json`도 함께 읽어 representative/diagnostic/domain count를 보강한다.
- 파일이 없으면 `unknown` 상태를 반환한다.

## 7. OUTPUT_QA_REPORT.md 파싱 규칙

현재 리포트 형식 예:

```text
- Status: `WARN`
- JSON files: `12`
- Excel files: `12`
- RUN_MANIFEST.json: `yes`
- Manifest inputs: `11`
- Manifest representative: `11`
- Manifest diagnostic: `1`
- Header/key mismatch: `0`
- Bad composite headers: `0`
```

파싱 원칙:

- backtick 안 숫자를 우선 추출
- `WARN`은 `warn`으로 normalize
- 누락 필드는 `unknown` 또는 0으로 둔다
- 파싱 실패해도 다운로드 가능성을 막지 않는다.

## 8. 다운로드 보안

Endpoint:

```text
GET /api/jobs/{job_id}/artifacts/{artifact_id}
```

규칙:

- artifact_id는 서버가 만든 매핑에서만 찾는다.
- 요청 path를 직접 파일 path로 해석하지 않는다.
- resolved path가 result_dir 내부인지 검사한다.
- directory는 다운로드 불가.
- `.env`, `.py`, `.db`, `.log`는 v1 다운로드 대상에서 제외한다.
- 허용 확장자는 `.md`, `.json`, `.xlsx`로 시작한다.

## 9. UI 표시 기준

Artifacts 화면:

- representative 파일을 먼저 표시
- diagnostic 파일은 별도 그룹으로 표시
- md/json/xlsx를 같은 stem 기준으로 묶어 표시
- QA/manifest/summary는 상단 고정 표시

QA 화면:

- `Status=OK`: 정상
- `Status=WARN`: 주의, 실패 아님
- `Status=FAIL`: 실패
- `Header/key mismatch > 0`: 강한 경고
- `Bad composite headers > 0`: 강한 경고
- `empty_tail_column`은 알려진 주의 항목으로 표시

## 10. 실패/예외 처리

- job 없음: `404 job_not_found`
- result_dir 없음: artifacts 빈 목록과 message 반환
- artifact 없음: `404 artifact_not_found`
- unsafe path: `400 unsafe_path`
- QA report 없음: `status=unknown`
- manifest JSON parse 실패: warning message 포함, artifacts는 계속 반환

## 11. 테스트 계획

- manifest 기반 representative artifact 추출
- diagnostic fallback artifact 표시
- QA report 숫자 파싱
- QA report 없는 job 처리
- path traversal 차단
- 허용되지 않는 확장자 다운로드 차단
- artifact_id가 임의 path로 동작하지 않는지 확인

## 12. 완료 기준

- artifacts endpoint가 결과 파일 목록 반환
- QA endpoint가 status/count/domain/warnings 반환
- md/json/xlsx 다운로드 가능
- unsafe path 차단
- manifest 없는 legacy output도 최소 목록 조회 가능
