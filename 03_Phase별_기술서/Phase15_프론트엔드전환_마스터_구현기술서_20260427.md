# Phase15 프론트엔드 전환 마스터 구현 기술서

작성일: 2026-04-27

대상 프로젝트: `Project/PJT_A/ps-docparser`

구현 방향: `FastAPI + React/Vite`

## 1. 목적

Phase15의 목적은 안정화된 `ps-docparser` 백엔드 파서 엔진을 로컬 웹앱에서 사용할 수 있게 만드는 것이다.

현재 백엔드는 CLI 중심이다. 프론트엔드에서 직접 호출하기 위해서는 HTTP API, 작업 상태 관리, 업로드 저장소, 결과 조회, 다운로드 계층이 필요하다.

Phase15는 제품형 v1 MVP를 목표로 하며, 범위는 다음으로 고정한다.

- PDF 업로드
- preset, engine, pages, BOM fallback 선택
- 파서 실행
- 상태와 로그 확인
- QA 요약 확인
- 결과 파일 다운로드

## 2. 백엔드 준비 상태

프론트엔드 착수 가능 기준은 이미 충족되었다.

검증 기준:

```text
git diff --check: PASS
focused tests: 93 passed
pytest: 753 passed, 6 skipped
coverage: 81.52%
real suite: Header/key mismatch 0, Bad composite headers 0
```

현재 백엔드가 제공하는 안정 계약:

- `RUN_MANIFEST.json`
- `RUN_SUMMARY.md`
- `OUTPUT_QA_REPORT.md`
- `domain`
- `role`
- `quality.status`
- `quality.warnings`
- `representative`
- `diagnostic`

결론:

- 파서 엔진 기준선은 프론트 MVP에 연결 가능하다.
- 웹 제품 백엔드는 아직 없으므로 FastAPI 어댑터가 필요하다.

## 3. 전체 아키텍처

```text
React/Vite UI
  -> FastAPI API
    -> Job Manager
      -> subprocess: python main.py ...
        -> existing ps-docparser CLI/pipelines
          -> output/ui_runs/<job_id>/result/
    -> Artifact Service
      -> RUN_MANIFEST.json
      -> RUN_SUMMARY.md
      -> OUTPUT_QA_REPORT.md
      -> md/json/xlsx artifacts
```

v1 원칙:

- 기존 파서 코어를 직접 리팩터링하지 않는다.
- 기존 CLI를 subprocess로 감싼다.
- output 계약은 백엔드 기준선의 manifest/QA 파일을 그대로 사용한다.
- 서버는 기본 `127.0.0.1` 로컬 바인딩이다.
- 인증, 멀티 유저, 원격 배포, DB 영속화는 제외한다.

## 6. 공통 API 계약

Phase15 v1 endpoint는 아래 7개로 고정한다.

```text
GET  /api/config
POST /api/jobs
GET  /api/jobs/{job_id}
POST /api/jobs/{job_id}/cancel
GET  /api/jobs/{job_id}/artifacts
GET  /api/jobs/{job_id}/artifacts/{artifact_id}
GET  /api/jobs/{job_id}/qa
```

공통 상태값:

```text
queued
running
succeeded
failed
canceled
```

공통 preset:

```text
auto
bom
estimate
pumsem
```

공통 engine:

```text
auto
zai
gemini
local
mistral
tesseract
```

`engine=auto`는 기존 CLI의 기본 동작을 따른다. API는 `--engine` 인자를 생략하고, CLI가 `.env`/config의 기본 엔진을 선택하게 한다.

BOM fallback:

```text
auto
always
never
```

## 7. 공통 데이터 흐름

1. 사용자가 React UI에서 PDF와 옵션을 선택한다.
2. React가 `POST /api/jobs`로 multipart 요청을 보낸다.
3. FastAPI가 `output/ui_runs/<job_id>/input/`에 업로드 파일을 저장한다.
4. Job Manager가 상태를 `queued`로 등록한다.
5. 백그라운드 실행기가 상태를 `running`으로 바꾸고 기존 CLI를 subprocess로 실행한다.
6. CLI 산출물은 `output/ui_runs/<job_id>/result/` 아래에 생성된다.
7. CLI 종료 후 `tools/analyze_outputs.py <result_dir>`를 실행한다.
8. 성공 시 상태는 `succeeded`, 실패 시 `failed`가 된다.
9. UI는 job 상태와 artifacts/QA endpoint를 polling한다.
10. 사용자는 결과 파일을 다운로드한다.

## 9. 공통 안전 규칙

- 업로드 확장자는 `.pdf`만 허용한다.
- 저장 파일명은 sanitize한다.
- artifact 다운로드는 job result 디렉토리 내부 파일만 허용한다.
- `..`, 절대 경로, 드라이브 루트 접근은 차단한다.
- `.env`와 API key는 응답, 로그, UI에 노출하지 않는다.
- `output/` 아래 산출물은 Git 추적 대상이 아니다.
- API 서버는 기본 `127.0.0.1`에서만 실행한다.
- Google Drive 동기화 경로에서는 `frontend/node_modules` 직접 설치를 피하고 `scripts/frontend_local.ps1`로 Drive 밖 로컬 작업 폴더에서 설치/빌드한다.
