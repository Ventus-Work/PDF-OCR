# Phase15-Step0 백엔드 기준선 잠금 상세 구현 기술서

작성일: 2026-04-27

대상 프로젝트: `Project/PJT_A/ps-docparser`

## 1. 목적

프론트엔드 구현 전에 백엔드 기준선을 명확히 고정한다.

Step0은 코드를 추가하는 단계가 아니라, 현재 안정화된 백엔드 상태를 프론트 구현의 출발점으로 인정하기 위한 잠금 단계다.

## 2. 구현 대상

대상:

- 현재 워크트리 변경분 감사
- 백엔드 기준선 문서
- 테스트 결과
- 실측 suite 결과
- 중복 PDF 샘플 처리 기준

직접 구현 대상이 아닌 것:

- FastAPI API 구현
- React UI 구현
- 파서 동작 변경
- PDF 샘플 이동/복구
- 커밋/푸시

## 3. 기준선으로 인정하는 상태

현재 기준선은 아래 결과를 기준으로 한다.

```text
git diff --check: PASS
focused tests: 93 passed
pytest: 753 passed, 6 skipped
coverage: 81.52%
real suite: Status WARN
Header/key mismatch: 0
Bad composite headers: 0
Quality warnings: empty_tail_column 1
```

`WARN`은 실패가 아니다. 현재 `empty_tail_column 1`은 알려진 품질 경고이며, JSON/Excel 구조 불일치나 복합 헤더 실패가 아니다.

## 5. 중복 PDF 샘플 처리 기준

대상 파일:

```text
01_OCR_결과물/53-83 OKOK.pdf
00_견적서_원본/53-83 OKOK.pdf
```

확인된 사실:

```text
deleted Git blob: b65cb4e2a993f5d87d6317fa87f7fa4804d3079e
00_견적서_원본 current file: b65cb4e2a993f5d87d6317fa87f7fa4804d3079e
```

정책:

- 로컬 실측 기준 샘플은 `00_견적서_원본/53-83 OKOK.pdf`로 둔다.
- `01_OCR_결과물/53-83 OKOK.pdf` 삭제 표시는 데이터 유실 위험이 아니라 중복 샘플 정리로 본다.
- 단, `00_견적서_원본/`은 `.gitignore` 대상이므로 Git 저장소에 PDF 샘플을 남길 필요가 있으면 별도 fixture 정책을 결정한다.

## 6. 커밋 단위 권장

커밋은 사용자가 명시 요청하기 전 실행하지 않는다.

권장 분리:

1. 백엔드 안정화 기준선 문서
2. BOM parser/output contract 변경
3. routing/fallback/manifest 변경
4. Gemini key rotation 변경
5. 중복 PDF 샘플 정리
6. Phase15 프론트 상세 기술서

프론트 구현 커밋은 위 기준선 잠금 뒤 별도 시작한다.

형식 검증:

```powershell

```

기준선 집중 테스트:

```powershell
pytest --no-cov tests/unit/validators/test_output_quality.py tests/unit/exporters/test_excel_exporter.py tests/unit/pipelines/test_document_pipeline.py tests/unit/pipelines/test_bom_pipeline.py tests/unit/utils/test_run_manifest.py tests/unit/tools/test_analyze_outputs.py tests/integration/test_main_cli_auto_route_smoke.py tests/integration/test_main_cli_bom_batch_aggregation_smoke.py tests/integration/test_main_cli_bom_fallback_smoke.py
```

전체 회귀:

```powershell
pytest
```

기존 output QA 확인:

```powershell
python tools/analyze_outputs.py output/실측테스트_20260427_baseline_lock
```

## 8. 실패/예외 처리

- `git diff --check` 실패: 문서/코드 whitespace 문제를 먼저 수정한다.
- focused tests 실패: 프론트 전환을 중단하고 백엔드 회귀를 먼저 고친다.
- 전체 `pytest` 실패: Phase15 구현을 시작하지 않는다.
- 실측 QA에서 header/key mismatch 또는 bad composite headers가 0이 아니면 기준선 미잠금으로 본다.
- PDF 샘플 정책이 다시 바뀌면 프론트 구현과 분리해 문서만 갱신한다.
